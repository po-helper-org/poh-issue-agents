---

## Использование

```
/validate-doc <any_context> <file_path>
```

**Параметры:**

- <any_context> – тезисная задача в произвольной форме
- `<file_path>` — путь к созданному файлу документации (например: /sa_documentation/docs/API-101.md)

## Примеры

```
/validate-doc провалидируй документ /sa_documentation/docs/API-101_OrderInfo.md
/validate-doc проверь системные требования /sa_documentation/FNR/FNR_1/system_requirements.md
```

## Важно

**Метод:** Сплошная проверка (Full Coverage) с учетом специфики документа.
**Токены:** Используй рефлексию в 3 итерации в рамках одного запроса.

## Инструкция для LLM

### Графовый контекст (complement-модель)

Граф дополняет repomix-output.xml, а не заменяет его. Используй оба источника одновременно:
- **repomix-output.xml** — полный текст кода (реализации, SQL, комментарии)
- **MCP-инструменты** (если sa-helper-graph подключён) — структура и связи

**Графовые запросы для валидации:**
- `graph_impact` — проверить, действительно ли сущность имеет указанные зависимости (CALLS, QUERIES, EXTENDS)
- `graph_call_chain` — проверить цепочку вызовов, описанную в документе
- `graph_schema` — сверить имена таблиц, классов, эндпоинтов с реальными в графе

Если MCP-инструменты недоступны — пропусти графовые запросы, верификация проводится через repomix-output.xml.

### Этап 1: Распознавание типа и сбор контекста

1. Прочитай целевой файл по пути `<file_path>`.
2. **Загрузи общие инструменты:** `repomix-output.xml`, `standard_mapping.md`, `sa_documentation/naming_conventions.md`, `/technical-documentation/resources/validation_checklist.md`
3. **Определи ТИП и догрузи специфику:**
   - **ТИП А (API Doc):** `.agent/skills/technical-documentation/SKILL.md`, `ideal_api_document.md`, `validation_checklist.md`.
   - **ТИП Б (Data Trace):** `.agent/skills/db-archeologist/SKILL.md`, `db_mapping_standards.md`, `ideal_data_trace.md`.
   - **ТИП В (System Requirements):** `/system-analyst-sysreq/SKILL.md`, `ideal_system_requirements.md`, `sysreq_validation_checklist.md`.

### ЭТАП 2: Доказательная база (Evidence Collection)

Прежде чем приступить к валидации, ты ОБЯЗАН найти в коде "Якоря Истины" (Truth Anchors):

**Для ТИПА А (API) и ТИПА Б (Data Trace):**

1. **Точка входа (Protocol Entry):** Найди артефакт, определяющий контракт интерфейса (Routing Table / Intercept Logic). Сверь HTTP-метод и способ парсинга данных.
2. **Слой ограничений (Validation Context):** Проанализируй логику проверки данных. Ищи скрытые зависимости (сквозь If/Else или Switch), влияющие на обязательность атрибутов.
3. **Узлы ветвления (Inference Nodes):** Идентифицируй архитектурные паттерны выбора реализации (Factory/Provider/Strategy).
4. **Инфраструктурный маппинг:** Соотнеси используемые в документе префиксы БД (из колонки Origin) с реальными подключениями в конфигурационных файлах проекта. Если есть расхождение или префикс отсутствует — статус 🔴.

**Для ТИПА В (System Requirements):**

1. **As-Is верификация:** Для каждого утверждения в разделе «Описание текущего поведения» найди подтверждающий код (файл:строка). Если утверждение не подтверждено — статус 🔴 или `[NEEDS_INVESTIGATION]`.
2. **Затрагиваемые компоненты:** Для каждой задачи проверь, что таблица «Затрагиваемые компоненты» содержит реальные файлы и строки из кодовой базы.
3. **Jira-ссылки:** Проверь, что ссылки на задачи в Jira указаны с номером и статусом.
4. **SQL-запросы:** Если в документе есть SQL-запросы (источники данных, приложения), проверь их корректность относительно реальной схемы БД.
5. **Диаграммы:** Проверь, что PlantUML-диаграммы синтаксически корректны и отражают описанную архитектуру.

### Этап 3: Сплошная проверка (4 Итерации рефлексии)

Каждая итерация должна завершаться внутренней проверкой соответствия **Hard Gates**.

#### Итерация 1: Точность Входов и Маршрутизации (The Gateway Gate) 🔴

**Для ТИПА А (API):**
- Проверь эндпоинты. Обязательно: `METHOD /path`. Запрещено: просто имя метода контроллера.
- Произвести верификацию URL, Итоговый URL в документе должен быть результатом явного сложения [Значение из Config] + [Значение из Interface/Annotation].
- Проверь "Источник в коде" для каждого параметра (напр. `$_GET`, `$_POST`).
- Для каждого параметра должен быть указан источник в транспорте (Body, Header, Path, Query). Совпадает ли это с логикой парсинга в коде, если нет, статус 🔴.
- Должны быть отражены зависимости полей, найденные на этапе "Evidence Collection".

**Для ТИПА Б (Data Trace):**
- Проверь, чтобы первая стрелка UML содержала реальный API-путь.

**Для ТИПА В (System Requirements):**
- Проверь структуру документа по шаблону `ideal_system_requirements.md`: наличие обязательных разделов (Введение, Общее описание с As-Is, План миграции, Функциональные блоки, Ревью, Риски и ограничения).
- Проверь метаданные в разделе «Общая информация»: наименование продукта, ответственные, Epic, БФТ, статус.
- Проверь нумерацию разделов — должна быть последовательной (1, 2, 3 / 3.1, 3.2 / 3.1.1, 3.1.2).
- Проверь таблицу «История изменений» — наличие хотя бы одной записи.

#### Итерация 2: Техническая достоверность (The Storage Gate) 🔴

**Для ТИПА А (API) и ТИПА Б (Data Trace):**
- **Объекты Хранения (State Stores):** Существуют ли указанные таблицы, индексы, очереди или ключи в исходном коде?
- **Происхождение (Origin Mapping):** Каждое техническое поле в документе должно иметь четкую привязку `Physical_Entity.Attribute`.
- **[CRITICAL] Full Entity Coverage:** Сверь список полей в документе с исходным DTO/Entity/$schema классом. Если в коде есть поля, которых нет в таблице — статус 🔴.
- Если в коде класса есть непустые поля, не отраженные в документе (даже если они не используются в бизнес-логике), агент ОБЯЗАН добавить их или обосновать пропуск.
- **Интеграционная Топология:** Указаны ли физические протоколы и конечные точки (URL/IP) внешних систем, извлеченные из конфигурационного контекста?
- **Словарь:** Сверь термины с `naming_conventions.md`. Если наткнулся на что-то новое – дай знать, возможно потребуется расширить naming_conventions.md
- **Якоря истины (Anchors of Truth):** В каждом документе обязателен раздел "Якоря истины" со ссылками на конкретные строки кода. Иначе статус 🔴.

**Для ТИПА В (System Requirements):**
- **As-Is Gate:** Каждое утверждение в разделе «Описание текущего поведения» должно иметь код-доказательство (файл:строка). Утверждения без доказательства — 🔴 или `[NEEDS_INVESTIGATION]`.
- **Task Gate:** Каждая задача в функциональных блоках должна содержать: обоснование, таблицу затрагиваемых компонентов (с файл:строка), критерии приёмки, зависимости.
- **Acceptance Gate:** Критерии приёмки должны быть конкретными и проверяемыми. «Работает корректно» без конкретики — 🔴.
- **NFR Gate:** Нефункциональные требования должны быть измеримыми (конкретные числа, имена классов, параметры). Оценочные формулировки («быстро», «надёжно») — 🔴.
- **Источники данных:** Если в задачах есть таблицы источников данных (SQL-запросы), проверить корректность имён таблиц и колонок относительно кодовой базы.
- **Маршрутизация:** Если есть таблицы маршрутизации (Сущность | Метод вызова), проверить что методы существуют в коде.

#### Итерация 3: Бизнес-логика и Визуализация (The Human-Readable Gate) 🔴/🟡

**Для всех типов:**
- **UML:** Все стрелки — на РУССКОМ. Описание должно объяснять **бизнес-смысл**, а технический метод писать в скобках.
- **Формат диаграмм:** Проверь, что НЕ используется Mermaid. Используй только PlantUML. Иначе статус СТРОГО 🔴.
- **Синтаксис и Зависимости:**
  1. Проверь все директивы `!include`.
  2. Проверь закрытие всех блоков (`@enduml`, `Boundary`, `Rel`).
  3. Если используется C4-PlantUML, проверь наличие `LAYOUT_WITH_LEGEND()`.
- **Визуализация:** Если диаграмма содержит внешние ссылки, которые могут быть недоступны — статус 🔴.
- **Терминология:** Соответствует ли лексика документа установленному Глоссарию проекта (Domain Glossary)?

**Для ТИПА А (API):**
- **[CRITICAL] Примеры данных:** В отчетах Data Trace ОБЯЗАТЕЛЕН блок `note` с примером структуры данных (JSON/Array). Если примера нет — статус СТРОГО 🔴.
- **Алгоритм:** Количество шагов в тексте должно совпадать с количеством шагов на Sequence-диаграмме.
- **Интеграции:** Вызовы внешних сервисов должны описываться не абстрактно, а с указанием точного физического протокола (SOAP, REST, gRPC) и эндпоинта из конфига.
- **Обработка Исключений:** Совпадает ли логика выброса исключений в коде с описанием ошибок в контракте?

**Для ТИПА В (System Requirements):**
- **Migration Gate:** План миграции должен содержать процедуру отката для каждого этапа. Без отката — 🔴.
- **Risks Gate:** Каждый риск должен иметь конкретную митигацию. Риски без митигации — 🔴.
- **Ограничения:** Раздел «Ограничения» должен перечислять, что решение НЕ покрывает, и компоненты backward compatibility.
- **Диаграммы:** Проверь наличие минимум двух диаграмм (компонентов + последовательности). Проверь корректность PlantUML-синтаксиса.

#### Итерация 3.1: Соответствие шаблонам и практикам аналитика (System Analyst Best Practise) 🔴

- **Определи ТИП документа и догрузи специфику:**
  - Для всех: `.visual_standards.md`
  - **ТИП А (API Doc):** `ideal_api_document.md`
  - **ТИП Б (Data Trace):** `ideal_data_trace.md`
  - **ТИП В (System Requirements):** `ideal_system_requirements.md` + `sysreq_validation_checklist.md`

- **Для ТИПА В (System Requirements):**
  - Пройдись по всем пунктам чек-листа `sysreq_validation_checklist.md`.
  - Проверь соответствие каждого раздела документа шаблону `ideal_system_requirements.md`.
  - Проверь, что все Hard Gates из чек-листа пройдены.
