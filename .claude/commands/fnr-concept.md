## Использование

```
/fnr-concept <path_to_task>
```

**Параметры:**

- `<path_to_task>` — путь к файлу `task.md` из предыдущего шага (`/fnr-task`).

## Примеры

```
/fnr-concept sa_documentation/FNR/FNR_1/task.md
```

## Важно

**Приоритет:** СТРОГОЕ соблюдение структуры из `examples/ideal_concept.md` и принципов из `SKILL.md`.

**Ожидаемый результат:** Файл `concept.md` в той же папке, что и `task.md` (например, `sa_documentation/FNR/FNR_1/concept.md`).

**Универсальность:** Команда не привязана к конкретному стеку, БД или бизнес-домену.

## Инструкция для LLM

### Графовый контекст (complement-модель)

Граф дополняет repomix-output.xml, а не заменяет его. Используй оба источника одновременно:
- **repomix-output.xml** — полный текст кода (реализации, SQL, комментарии)
- **MCP-инструменты** (если sa-helper-graph подключён) — структура и связи

**Графовые запросы для данной команды:**
- `graph_export` — обзор структуры проекта для оценки зоны воздействия
- `graph_impact` — анализ связей затрагиваемых компонентов

Если MCP-инструменты недоступны — пропусти графовые запросы, продолжай с repomix-output.xml.

### Этап 1: Загрузка роли

1. Прочитай `/solution-designer/SKILL.md` — твоя персона и принципы.
2. Загрузи шаблон из `examples/ideal_concept.md`.

### Этап 2: Сбор контекста проекта

1. Прочитай файл `<path_to_task>` — это постановка задачи.

2. Проверь существование `sa_documentation/repomix-output.xml`.
   - Если существует — загрузи как первичный источник кода.
   - Если нет — предупреди: "repomix-output.xml не найден. Рекомендуется запустить `/context-gen`. Продолжаю с поиском по живому коду."

3. Прочитай артефакты из `sa_documentation/`:
   - `naming_conventions.md` (если есть) — словарь терминов проекта.
   - Архитектурные документы — не предложить концепт, противоречащий принятой архитектуре.
   - Существующие документы — не дублировать.

4. Поиск по живому коду — для проверки конкретных гипотез.

### Этап 3: Анализ проблемы и ограничений

1. Выдели из task.md:
   - Корень проблемы
   - Зону воздействия
   - Ограничения (что нельзя трогать, бюджет времени)

2. Проанализируй кодовую базу (repomix / живой поиск):
   - Какие механизмы уже существуют и могут быть использованы
   - Какие компоненты жёстко связаны и трудно меняются
   - Где есть «пространство для манёвра»

### Этап 4: Генерация концептов

Сгенерируй 3-5 концептов разного уровня (обязательно покрытие минимум 3 категорий):

| Категория | Обязательность |
|-----------|---------------|
| Правильное | Обязательно |
| Прагматичное | Обязательно |
| Быстрый костыль | Обязательно |
| Креативный хак | Опционально |
| «Не трогай» | Если уместно |

Для каждого концепта:

- Опиши суть (2-3 предложения)
- Укажи конкретные файлы/компоненты, которые нужно менять (из repomix)
- Перечисли плюсы и минусы
- Оцени усилия (часы/дни)
- Оцени риски

### Этап 5: Сохранение

1. Создай файл `concept.md` рядом с `task.md` (в той же папке FNR_<N>).
2. Заполни по шаблону `ideal_concept.md`.

### Этап 6: Завершение

Выведи: "Концепты решений сохранены в `sa_documentation/FNR/FNR_<N>/concept.md`. Следующий шаг: `/fnr-system-requirements sa_documentation/FNR/FNR_<N>/concept.md`"
