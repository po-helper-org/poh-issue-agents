description: Генерация документа системных требований (System Requirements) на основе утверждённого концепта решения
---

## Использование

```
/fnr-system-requirements <path_to_concept>
```

**Параметры:**

- `<path_to_concept>` — путь к файлу `concept.md` с вердиктом дебатов (`/fnr-debate`).

## Примеры

```
/fnr-system-requirements sa_documentation/FNR_1/concept.md
```

## Важно

**Приоритет:** СТРОГОЕ соблюдение структуры из `examples/ideal_system_requirements.md` (шаблон в стиле Confluence).

**Ожидаемый результат:** Документ сохраняется в папку FNR: `sa_documentation/FNR/FNR_<N>/system_requirements.md`.

**Универсальность:** Команда не привязана к конкретному стеку, БД или бизнес-домену.

## Инструкция для LLM

### Графовый контекст (complement-модель)

Граф дополняет repomix-output.xml, а не заменяет его. Используй оба источника одновременно:
- **repomix-output.xml** — полный текст кода (реализации, SQL, комментарии)
- **MCP-инструменты** (если sa-helper-graph подключён) — структура и связи

**Графовые запросы для данной команды:**
- `graph_schema` — обзор структуры для заполнения As-Is
- `graph_export` — детали компонента для затрагиваемых компонентов
- `graph_impact` — анализ зависимостей для обоснования задач

Если MCP-инструменты недоступны — пропусти графовые запросы, продолжай с repomix-output.xml.

### Этап 1: Загрузка роли

1. Прочитай `/system-analyst-sysreq/SKILL.md` — твоя персона и принципы работы.
2. Загрузи **основной шаблон** документа из `examples/ideal_system_requirements.md` — это шаблон в стиле Confluence.
3. Загрузи чек-лист валидации из `resources/sysreq_validation_checklist.md`.

### Этап 2: Сбор контекста проекта

1. Прочитай `<path_to_concept>` — концепты решений + вердикт дебатов.
2. Прочитай `task.md` из той же папки — постановка проблемы.
3. Извлеки из вердикта: выбранный концепт, модификации из дебатов.

4. Проверь существование `sa_documentation/repomix-output.xml`.
   - Если существует — загрузи как первичный источник кода.
   - Если нет — предупредить: "repomix-output.xml не найден. Рекомендуется запустить `/context-gen`. Продолжаю с поиском по живому коду."

5. Прочитай артефакты из `sa_documentation/`:
   - `naming_conventions.md` — терминология.
   - Существующие документы — наследовать стиль, не дублировать.
   - Поиск по живому коду — для точной детализации FR.

### Этап 3: Формирование документа по шаблону Confluence

Заполни документ **СТРОГО** по шаблону `ideal_system_requirements.md`:

#### Структура документа (обязательные разделы):

1. **Введение (раздел 1)** — общая информация (метаданные в таблице) + термины и определения + ссылки + история изменений.
2. **Общее описание (раздел 2)**:
   - **As-Is** — описание текущего поведения с код-доказательствами (ключевые компоненты + ограничения текущего решения).
   - Архитектурное решение (ссылка).
   - Диаграмма компонентов (PlantUML).
   - Схема последовательности (PlantUML).
3. **План миграции (раздел 3)** — этапы внедрения (PlantUML), таблица этапов с откатами, критерии готовности.
4. **Функциональные требования — Backend / БД / API (раздел 4)** — серверные задачи:
   - **Что попадает:** изменения в БД (миграции, таблицы, views), backend-логика (контроллеры, сервисы, роутинг), API-эндпоинты, фоновые процессы (cron, очереди), интеграции между backend-сервисами, поисковые движки.
   - Каждая задача по шаблону: метаданные, описание, обоснование, затрагиваемые компоненты, критерии приёмки, зависимости.
   - **Для API-задач** дополнительно: перечень эндпоинтов (таблица), формат ответа (JSON-примеры), статусы (таблица).
   - **Для задач обработки данных** дополнительно: маршрутизация (таблица), источники данных (таблица).
   - **Нефункциональные требования** (если применимы) — конкретные, измеримые.
5. **Требования к интерфейсам — Frontend / UI (раздел 5)** — клиентские задачи:
   - **Что попадает:** верстка экранов и UI-компонентов, изменения в SSR/CSR, клиентские интеграции с API, микроразметка (schema.org, Open Graph), клиентский роутинг.
   - **Для задач по верстке (5.1):** описание требований + критерии приёмки.
   - **Для задач по UI-интеграциям (5.2):** расширенный шаблон с эндпоинтами, форматом ответа, NFR, затрагиваемыми компонентами.
   - **Если документ не содержит UI-изменений** — раздел 5 опускается с пометкой «Не применимо».
6. **Ревью требований** — таблица согласования (аналитик, разработка Backend, разработка Frontend, тестирование).
7. **Риски и ограничения** — таблица рисков (ID | Риск | Вероятность | Влияние | Митигация) + нумерованный список ограничений.
8. **Приложения** (опционально) — справочные материалы: SQL-скрипты, маппинги, шпаргалки.

#### Правило разделения Backend и Frontend

**Принцип:** каждая задача относится либо к Backend (раздел 4), либо к Frontend (раздел 5).

- **Backend (раздел 4):** затрагивает серверный код, БД, API, Sphinx, фоновые процессы, межсервисные интеграции.
- **Frontend (раздел 5):** затрагивает клиентский код, верстку, UI-логику, микроразметку, клиентский роутинг.
- **Смежные фичи:** если одна фича требует изменений и на backend, и на frontend — создаются **две** задачи:
  1. Backend-задача в разделе 4 (создание API, изменение БД).
  2. Frontend-задача в разделе 5 (интеграция с API, верстка).
  3. Связь через раздел «Зависимости» (frontend-задача зависит от backend-задачи).
- **Только Backend:** если документ не содержит UI-изменений — раздел 5 опускается с пометкой «Не применимо».
- **Только Frontend:** если документ не содержит backend-изменений — раздел 4 опускается с пометкой «Не применимо».

#### Правила заполнения:

- Метаданные каждой задачи — в таблице (Ответственный за тех. реализацию + Задача на разработку).
- Jira-ссылки — с номером и статусом задачи.
- Все нумерованные списки — арабскими цифрами, подпункты — a, b, c.
- Критерии приёмки — конкретные проверяемые условия (как QA проверит).
- NFR — конкретные параметры (имена классов, размеры пулов, таймауты).
- JSON-примеры — в блоках кода с разметкой `json`.
- SQL-примеры — в блоках кода с разметкой `sql`, с комментариями.
- Диаграммы — PlantUML (Mermaid запрещён), надписи на русском.

### Этап 4: Визуализация

Сгенерируй диаграммы СТРОГО в формате **PlantUML** (Mermaid запрещён):

1. **Диаграмма компонентов** — компоненты системы и взаимосвязи.
2. **Схема последовательности** — порядок взаимодействия компонентов.
3. **План миграции** — этапы внедрения (activity diagram).

Все надписи на стрелках — на русском языке (бизнес-смысл). Технические имена — в скобках.

### Этап 5: Самопроверка и сохранение

1. Пройдись по чек-листу `resources/sysreq_validation_checklist.md`.
2. Убедись, что документ соответствует шаблону `ideal_system_requirements.md`:
   - Есть разделы: Введение, Общее описание (с As-Is), План миграции, Функциональные блоки, Ревью, Риски и ограничения.
   - Каждая задача имеет метаданные, описание, обоснование, затрагиваемые компоненты, критерии приёмки, зависимости.
   - API-задачи имеют эндпоинты, формат ответа, статусы.
   - Диаграммы в PlantUML.
   - Риски с митигацией + ограничения.
3. Убедись, что каждое утверждение подтверждено код-доказательством.
4. Сохрани документ в `sa_documentation/FNR/FNR_<N>/system_requirements.md` (рядом с task.md и concept.md).
5. Выведи: "Системные требования готовы. Следующий шаг: `/validate-doc <путь_к_файлу>`"
