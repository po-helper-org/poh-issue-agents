## Использование

```
/arch-gen <context>
```

**Параметры:**

- `<context>` — описание того, что хочет сделать аналитик для документирования сервиса и описания архитектуры

## Примеры

```
/arch-gen Я, как системный аналитик, должен провести реверс-инжиниринг текущего проекта. Мне нужно полностью задокументировать проект: составить C4-диаграмму (до уровня C3), чтобы понимать все бизнес-контексты, компоненты и способы их взаимодействия друг с другом. Я должен понять: 
1. Какие бизнес-контексты есть в проекте?
2. Какие компоненты есть в каждом бизнес-контексте?
3. Какие способы взаимодействия есть между компонентами?
4. Какие данные передаются между компонентами?
5. Какие протоколы используются для взаимодействия между компонентами?
6. Какие внешние системы взаимодействуют с проектом?
7. Какие данные передаются между внешними системами и проектом?
8. Какие протоколы используются для взаимодействия между внешними системами и проектом?
```

## Важно

**Приоритет:** СТРОГОЕ соблюдение структуры из `examples/ideal_api_document.md` и нотации из `resources/standard_mapping.md`.
**Ожидаемый результат**:  Все документы, создаваыемые в рамках это команды, должны сохраняться по следующему пути `sa_documentation/architecture`. Если папки `architecture` нет – создай ее.

## Инструкция для LLM

### Графовый контекст (complement-модель)

Граф дополняет repomix-output.xml, а не заменяет его. Используй оба источника одновременно:
- **repomix-output.xml** — полный текст кода (реализации, SQL, комментарии)
- **MCP-инструменты** (если sa-helper-graph подключён) — структура и связи

**Графовые запросы для данной команды:**
- `graph_schema` — обзор структуры проекта (классы, эндпоинты, таблицы, внешние сервисы)
- `graph_export` — экспорт графа (текст или Mermaid-диаграмма)
- `graph_impact` — зависимые компоненты класса

Если MCP-инструменты недоступны — пропусти графовые запросы, продолжай с repomix-output.xml.

### Этап 0: Загрузка роли Архитектора

1. **Загрузи навык:** `.../skills/architecture/SKILL.md`**. Это твоя основная личность для этой задачи.

### Этап 1: Инвентаризация и Маппинг (Chain of Thought)

#### Формирование контекста

0. Дополнительный сбор контекста: Используя resources/discovery_rules.md, проанализируй repomix-output.xml и сформируй файл resources/project_taxonomy.md. Распредели классы по бизнес-доменам (Catalog, Payment и т.д.)
1. Прежде чем генерировать документацию, проведи глубокий анализ `repomix-output.xml`

#### Чероновик связей

1. **Выдели Интеграции:** Найди все вызовы внешних API (Payment Gateway, Auth Provider и т.д.). Сопоставь их с классами проекта.
2. **Распредели по Доменам:** Используй `resources/project_taxonomy.md` (если есть) или логику нейминга, чтобы разнести классы по бизнес-доменам проекта.
3. **Определи БД:** Раздели запросы по инстансам БД, если конфигурация проекта использует несколько подключений.

### Этап 2: Проверка соответствия "Золотому стандарту"

Сверь планируемую схему с `examples/ideal_c4_l3.puml`:

- Выделен ли кэширующий компонент как отдельная сущность?
- Выделены ли компоненты управления блокировками/резервированиями?
- Связаны ли Gateway-компоненты с внешними сервисами?

### Этап 3: Генерация PlantUML

Используй следующую структуру:

1. **Boundaries:** - `System_Boundary(app, "Application (<стек_проекта>)")`
   - Внутри него под-границы: `Boundary(business, "Business Logic Layer")` и `Boundary(data_access, "Data & Integration Layer")`.
2. **External:** - `System_Boundary(microservices, "Микросервисы и Внешние интеграции")` — здесь должны быть внешние сервисы.
3. **Databases:** - `System_Boundary(databases, "Хранилища данных")` — поисковый движок, кэш, основная БД (определяются из конфигурации проекта).

### Этап 4: Финализация связей

Проверь, что связи `Rel` содержат протоколы (REST, SOAP, SQL, TCP).
