## Использование

```
/data-trace <entity_name_or_field>
```

## Примеры

**Пример №1**

```
/data-trace Show
```

**Пример №2**

```
/data-trace атрибут Show.title
```

**Пример №3**

```
/data-trace данные, отображаемые на странице интерфейса по ссылке `{url_example}`
```

## Важно

**Приоритет:** СТРОГОЕ соблюдение структуры из `.agent/skills/architecture/examples/ideal_data_trace.md` и нотации из `resources/standard_mapping.md`.
**Ожидаемый результат**:  Все документы, создаваемые в рамках это команды, должны сохраняться по следующему пути `sa_documentation/data_trace`. Если папки `data_trace` нет – создай ее.

## Инструкция для LLM

### Графовый контекст (complement-модель)

Граф дополняет repomix-output.xml, а не заменяет его. Используй оба источника одновременно:
- **repomix-output.xml** — полный текст кода (реализации, SQL, комментарии)
- **MCP-инструменты** (если sa-helper-graph подключён) — структура и связи

**Графовые запросы для данной команды:**
- `graph_call_chain` — трассировка цепочки вызовов от сущности (указать entity_name и depth)
- `graph_impact` — анализ зависимых компонентов сущности
- `graph_export` — подграф вокруг сущности (экспорт связей)

Если MCP-инструменты недоступны — пропусти графовые запросы, продолжай с repomix-output.xml.

### ПРАВИЛО №1: ФОРМАТ ВИЗУАЛИЗАЦИИ (CRITICAL)

- **ТОЛЬКО PlantUML.** Использование Mermaid или других форматов СТРОГО ЗАПРЕЩЕНО.
- **ЯЗЫК:** Все комментарии на стрелках и в примечаниях диаграммы должны быть на **РУССКОМ ЯЗЫКЕ**.
- **READABILITY:** Все участники (participants) и сообщения на стрелках должны иметь человекочитаемые описания на русском языке. Технические названия методов/классов указываются в скобках.
- **NOTATION:** Обязательно использование семантических сущностей: `actor` (пользователь), `database` (БД/Кэш), `activate/deactivate` (активация), `alt/else` (обработка ошибок и условий).

### Этап 1: Поиск точек входа (Omnichannel Entry Points)

1. Найди **ВСЕ возможные точки входа** в систему для данной сущности. Ты ОБЯЗАН проверить наличие:
   - Web-контроллеров (SSR / HTML рендеринг)
   - REST API эндпоинтов (для SPA / Frontend)
   - Mobile API (для мобильных приложений)
   - CLI-скриптов / Cron задач или обработчиков очередей (RabbitMQ/Kafka)
   Не останавливайся на первом найденном методе. Зафиксируй форматы данных для каждого из каналов отдельно.
2. **Route Mapping:** Обязательно найди и укажи, как внешние конечные адреса (Endpoints) маппятся на методы контроллеров через конфигурационные файлы роутинга (например: `routes`, `url_routes`, `api-gateway`).

### Этап 2: Трассировка по слоям (The Chain)

Пройди по цепочке вызовов, используя `repomix-output.xml`:

1. **Validation Layer:** Какие проверки проходит это поле?
2. **Business Logic:** Как поле трансформируется? (Напр.: "Массив сырых данных превращается в DTO объект").
3. **Integration Layer:** Передается ли это поле во внешние системы? Если да — найди API-контракт, укажи точный транспорт (REST/SOAP/AMQP) и физический адрес ресурса (URL/WSDL/Топик), извлеченный из локальных конфигов проекта.
4. **Data Access:** В какую таблицу и колонку в итоге попадает значение? Используй нотацию `[DB_NAME].[SCHEMA].[TABLE].[COLUMN]`.

### Этап 3: Визуализация (PlantUML)

Создай Sequence Diagram, которая показывает движение этого конкретного поля данных через все компоненты системы. Сделай диаграмму максимально понятной для аналитиков, дополняя её пояснительными `note`.

### Этап 4: Результат

Выведи отчет "Data Lineage Report":

- **Entry Points:** (Таблица: Назначение (Web/API/Mobile), Внешний URL, Backend Метод)
- **Transformations:** (Что с ним происходило на уровне Service)
- **Destination:** (Где сохранилось: Таблица, Колонка, Тип данных)
- **Side Effects:** (Где еще используется, например, инвалидация кэша в Redis)
