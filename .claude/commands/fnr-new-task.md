## Использование

```
/fnr-new-task <description>
```

**Параметры:**

- `<description>` — свободное текстовое описание проблемы, бага, запроса заказчика или идеи.

## Примеры

```
/fnr-new-task После cron-импорта данные, созданные вручную в web_db, пропадают. Нужно понять причину и описать проблему.
```

```
/fnr-new-task Сервис авторизации не поддерживает одновременные сессии. Пользователь логинится на втором устройстве — первого выкидывает.
Контекст: /auth/service.py, /auth/session_manager.py
```

```
/fnr-new-task Заказчик хочет, чтобы поиск работал с опечатками. Сейчас только точное совпадение.
```

## Важно

**Приоритет:** СТРОГОЕ соблюдение структуры из `examples/ideal_task.md` и принципов из `SKILL.md`.

**Ожидаемый результат:** Файл `sa_documentation/FNR/FNR_<N>/task.md`, где `<N>` — следующий доступный номер. Если `sa_documentation/FNR/FNR_1/` уже есть — создать `FNR_2/` и т.д.

**Универсальность:** Команда не привязана к конкретному стеку, БД или бизнес-домену.

## Инструкция для LLM

### Графовый контекст (complement-модель)

Граф дополняет repomix-output.xml, а не заменяет его. Используй оба источника одновременно:
- **repomix-output.xml** — полный текст кода (реализации, SQL, комментарии)
- **MCP-инструменты** (если sa-helper-graph подключён) — структура и связи

**Графовые запросы для данной команды:**
- `graph_schema` — обзор структуры проекта для понимания контекста
- `graph_impact` — поиск зависимостей проблемного компонента
- `graph_call_chain` — трассировка вызовов для поиска корня проблемы

Если MCP-инструменты недоступны — пропусти графовые запросы, продолжай с repomix-output.xml.

### Этап 1: Загрузка роли

1. Прочитай `/problem-analyst/SKILL.md` — твоя персона и принципы.
2. Загрузи шаблон из `examples/ideal_task.md`.
3. Загрузи чек-лист из `resources/task_validation.md`.

### Этап 2: Сбор контекста проекта

1. Проверь существование `sa_documentation/repomix-output.xml`.
   - Если существует — загрузи как первичный источник кода.
   - Если нет — предупреди: "repomix-output.xml не найден. Рекомендуется запустить `/context-gen` для оптимизации контекста. Продолжаю с поиском по живому коду."

2. Прочитай артефакты из `sa_documentation/`:
   - `naming_conventions.md` (если есть) — словарь терминов проекта.
   - `tasks.md` (если есть) — контекст текущих задач.
   - Существующие документы (docs/, architecture/, data_base_docs/ и т.д.) — не дублировать.

3. Поиск по живому коду — только для уточнений, не покрытых repomix или артефактами.

### Этап 3: Деконструкция запроса

1. Разбери `<description>` на:
   - **Проблему:** что сломано / чего не хватает.
   - **Контекст:** в каком окружении, какие ограничения.
   - **Ожидание:** что пользователь хочет получить.

2. Классифицируй тип задачи: баг / рефакторинг / новая фича / оптимизация / архитектурное изменение.

### Этап 4: Исследование кодовой базы

На основе repomix-output.xml и/или поиска по коду:

1. Найди компоненты, реализующие текущее поведение.
2. Найди конфигурации, определяющие ограничения.
3. Определи зависимости между компонентами.
4. Найди корень проблемы — не симптом, а причину.

### Этап 5: Формирование task.md

1. Если `sa_documentation/` не существует — создай.
2. Если `sa_documentation/FNR/` не существует — создай.
3. Определи следующий доступный номер FNR (проверь `sa_documentation/FNR/FNR_*/`, если папок нет — начни с `FNR_1`).
4. Создай папку `sa_documentation/FNR/FNR_<N>/`.
5. Создай файл `task.md` по шаблону `ideal_task.md`.
6. Каждое утверждение подтверди код-доказательством (файл:строка).
7. НЕ предлагай решений — только диагноз.

### Этап 6: Завершение

Выведи: "Задача FNR-<N> описана в `sa_documentation/FNR/FNR_<N>/task.md`. Следующий шаг: `/fnr-concept sa_documentation/FNR/FNR_<N>/task.md`"
