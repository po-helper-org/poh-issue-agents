## Использование

```
/context-gen <context>
```

**Параметры:**

- `<context>` — описание цели того, что хочет сделать аналитик для документирования сервиса

## Примеры

**Пример №1**

```
/context-gen Я, как системный аналитик, должен провести revese-engineering текущего проекта. Мне нужно полностью задокументировать проект:
- Составить C4-диаграмму (до уровня C3), описать основные компоненты сервиса, описать базу данных, 
- Описать имеющиеся API-интерфейсы, которые предоставляет сервис, (один API = одна задача = один документ)
- Описать интеграционные взаимодействия данного сервиса с другими сервисами (одна интеграция = одна задача = один документ). 
- Описать способы взаимодействия данного проекта с базой данных (если есть)
```

**Пример №2**

```
/context-gen Я хочу задокументировать архитектуру проектап и создать C4-диаграмму.
```

## Важно

## Величина контекста

**Целевой лимит:** $800,000$ токенов

**Критический порог:** $1,200,000$ токенов

## Правила фильтрации контекста

Исключать из анализа:

1. **Boilerplate:** Автогенерируемые файлы миграций, логи, временные файлы.
2. **Vendor Bloat:** Папки `node_modules`, `vendor`, `dist`.
3. **Test Data:** Моки и фикстуры, если задача не касается тестирования.
4. **Assets:** Картинки, шрифты, скомпилированный CSS/JS.

## На выходе ожидается

- Созданная папка `sa_documentation`
- Файл `repomix-output.xml` (Контекст проекта)
- Файл `.repomixignore`
- Файл `sa_documentation/tasks.md` (Список декомпозированных задач для документирования)
- Файл `sa_documentation/naming_conventions.md` (Словарь соответствия кода и бизнеса)

**Иные файлы не создавать!**

## Инструкция для LLM

### Этап 0: Подготавливаем директории для работы

1. Создай папку .../sa_documentation.

### Этап 1: Инициализация и первичный замер

1. Запусти ```repomix --no-files```
2. Проанализируй раздел ```Pack Summary``` (Total Tokens) и ```Top 5 Files by Token Count```

### Этап 1.5: Графовая фильтрация (complement-модель)

Если MCP-инструменты доступны (sa-helper-graph подключён):
1. Вызови `graph_select_files` с параметром `task_description="<описание задачи пользователя>"` — инструмент вернёт JSON-массив релевантных путей
2. Из JSON-массива путей сформируй glob-паттерн для repomix: оберни список в фигурные скобки через запятую. Пример: для `["src/controllers/UserController","src/services/OrderService"]` получится `"{{src/controllers/UserController,src/services/OrderService}}"`
3. При запуске repomix на Этапе 2 передай этот glob в флаг `--include`, например: `repomix --style xml --output repomix-output.xml --include "{src/controllers/UserController,src/services/OrderService}"`
4. Если MCP-инструмент недоступен или вернул пустой массив — продолжай без фильтрации (repomix упакует все файлы без флага --include)

### Этап 2: Петля оптимизации (Intelligent Trimming Loop)

**Если** ```Total Tokens``` > $1,000,000$:

1. Изучи список самых тяжелых файлов в Summary.
2. Определи, являются ли они бизнес-логикой. Если это библиотеки, логи, тяжелые стили или сгенерированные файлы — Создай или дополни .repomixignore, добавив их пути в .repomixignore.
3. **ВАЖНО:** Если тяжелым является файл с бизнес-логикой (например, сервисный файл на 200k токенов), не игнорируй его, но ищи другие пути сокращения (например, исключи папки с тестами или документацией внутри проекта).
4. Повтори запуск repomix --summary-only и проверь новый результат.
5. Продолжай цикл до тех пор, пока объем не станет < $1,000,000$ токенов или пока не останутся только критически важные файлы.

### Этап 3: Финальная сборка

1. Только после прохождения проверки лимитов запусти: ```repomix --style xml --output repomix-output.xml```.
2. Если даже после очистки объем $> 1,200,000$ токенов, выведи предупреждение пользователю: "Внимание: проект очень крупный. Планирование может быть менее точным".
3. Файл должен быть перенесен в ранее созданную папку /sa_documentation/repomix-output.xml и удален из корня проекта, где он изначально создавался

#### Этап 3.1: Картирование публичного API

1. Найди файлы конфигурации роутинга.
2. Составь таблицу: `Внешний URL` -> `Внутренний контроллер/метод`.
3. Сравни этот список со списком файлов в `repomix-output.xml`.
4. Если обнаружен "скрытый" эндпоинт (как announcement -> ads), принудительно добавь его в `naming_conventions.md` и выдели под него задачу в `tasks.md`.

### Этап 4: Интеллектуальное планирование

1. Прочитай итоговый ```/sa_documentation/repomix-output.xml```
2. Используя универсальные паттерны из `resources/architecture_patterns.md`, определи ключевые слои (Controllers, Services, DAO).
3. Создай файл ```/sa_documentation/tasks.md```, распределим задачи так, чтобы файлы в одной задаче суммарно не превышали $100,000$ токенов.

#### Правило классификации задач по типу документа

Для каждой задачи определи её **тип** и соответствующую подпапку:

| Тип задачи | Признаки | Подпапка | Префикс файла |
|---|---|---|---|
| **API** | Описание REST/GraphQL/SOAP эндпоинтов, контроллеров, роутинга запросов | `api` | `api_` |
| **DataTrace** | Трассировка данных от входа до хранилища, lineage полей, анализ трансформаций | `data_trace` | `data_trace_` |
| **Architecture** | C4-диаграммы, описание компонентов, интеграционные взаимодействия между сервисами | `architecture` | `architecture_` |

**Формат имени файла:** `<префикс>_<человекочитаемое_название>.md` (например: `api_show_listing.md`, `data_trace_order_status.md`, `architecture_c4_overview.md`)

Каждая задача должна иметь следующий вид:

```
[ID] Название задачи (например: "Чтение данных по Show")

Бизнес-цель: (Описать понятным языком, что делает этот узел).

Тип задачи: (API | DataTrace | Architecture)

Необходимый контекст (Файлы):
**Global Context:** Какие архитектурные принципы или общие файлы (конфиги, интерфейсы) должны быть в памяти.

**Local Context:** Список конкретных путей к файлам, которые содержат логику этой задачи.
[ ] path/to/config (Global)
[ ] path/to/service (Local)
[ ] path/to/entity (Data)

**Связи для проверки:** (На что обратить внимание, чтобы не соврать, например: "Проверить маппинг поля status_id в коде vs status_name в БД").

**Шаблон результата:** sa_documentation/<подпапка>/<префикс>_<название>.md
```

# Этап 5: Формирование словаря системы (Naming Conventions)

1. На основе анализа кода в XML выдели ключевые бизнес-сущности.
2. «Скан конфигов: Перед формированием таблицы маппинга, сопоставь публичные URL-пути из файлов конфигурации с именами контроллеров. Если название в URL (напр. announcement) отличается от названия в коде (Ads), в словарь должны попасть оба термина с пометкой 'Public Alias'».
3. Создай файл `/sa_documentation/naming_conventions.md`.
4. Заполни его таблицей соответствия. Используй логику:
   - **Понятие:** Как это называть в документах (напр., "Заказ").
   - **Технический термин:** Как это называется в коде (напр., `STOrder`, `UserBasket`).
   - **Описание:** Краткая суть сущности.
5. Добавь раздел "Запрещенные синонимы", чтобы избежать путаницы (например, если в коде "Client", а по бизнесу это "User").
