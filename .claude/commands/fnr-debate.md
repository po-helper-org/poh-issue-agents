## Использование

```
/fnr-debate <path_to_concept>
```

**Параметры:**

- `<path_to_concept>` — путь к файлу `concept.md` из предыдущего шага (`/fnr-concept`).

## Примеры

```
/fnr-debate sa_documentation/FNR/FNR_1/concept.md
```

## Важно

**Метод:** Три раунда структурированных дебатов между двумя ролями: Архитектор и Адвокат Дьявола.

**Ожидаемый результат:** Дебаты и вердикт дописываются в конец файла `concept.md`.

**Универсальность:** Команда не привязана к конкретному стеку, БД или бизнес-домену.

## Инструкция для LLM

### Графовый контекст (complement-модель)

Граф дополняет repomix-output.xml, а не заменяет его. Используй оба источника одновременно:
- **repomix-output.xml** — полный текст кода (реализации, SQL, комментарии)
- **MCP-инструменты** (если sa-helper-graph подключён) — структура и связи

**Графовые запросы для данной команды:**
- `graph_call_chain` — проверить цепочку вызовов, упомянутую в аргументе
- `graph_impact` — проверить реальные зависимости сущности
- `graph_schema` — сверить утверждения со структурой проекта

Если MCP-инструменты недоступны — пропусти графовые запросы, верификация проводится через repomix-output.xml.

### Этап 1: Загрузка роли

1. Прочитай `/architectural-debate/SKILL.md` — правила проведения дебатов.
2. Прочитай `resources/debate_rules.md` — формат и ограничения раундов.

### Этап 2: Сбор контекста проекта

1. Прочитай файл `<path_to_concept>` — концепты решений.
2. Найди и прочитай соответствующий `task.md` в той же папке.

3. Проверь существование `sa_documentation/repomix-output.xml`.
   - Если существует — загрузи как первичный источник кода.
   - Если нет — продолжай с поиском по живому коду.

4. Прочитай артефакты из `sa_documentation/`:
   - Архитектурные документы — сверка концептов с принятой архитектурой.
   - `naming_conventions.md` — терминология.
   - Существующие документы — контекст.

5. Поиск по живому коду — для проверки утверждений в раундах.

### Этап 3: Проведение дебатов (3 раунда)

Проводи дебаты строго по правилам из `resources/debate_rules.md`. Каждый аргумент подтверждай ссылкой на код.

**Раунд 1 — Защита:**

- Архитектор выбирает лучший концепт и защищает его.
- Адвокат Дьявола атакует: находит слабые места, риски, скрытые проблемы.

**Раунд 2 — Пересмотр:**

- Архитектор учитывает критику: усиливает аргументацию или пивотит на другой концепт.
- Адвокат проверяет: «А что если условие X изменится через полгода?».

**Раунд 3 — Вердикт:**

- Обе стороны формулируют итоговую позицию.
- Формируется вердикт.

### Этап 4: Формирование вердикта

Вердикт может быть любым:

| Вердикт | Когда |
|---------|-------|
| Концепт принят без изменений | Обе стороны согласны |
| Концепт принят с модификациями | Адвокат нашёл слабые места, Архитектор предложил усиления |
| Пивот на другой концепт | Лучший концепт не выдержал критику |
| Все концепты забракованы | Ни один концепт не прошёл проверку |
| Проблема не требует технического решения | Это процессная/организационная проблема |

### Этап 5: Сохранение

1. Допиши дебаты и вердикт в конец файла `<path_to_concept>`.
2. НЕ перезаписывай существующее содержание — только дополнение.

### Этап 6: Завершение

Выведи:

- Если вердикт = концепт принят: "Вердикт: <выбранный концепт>. Следующий шаг: `/fnr-system-requirements <path_to_concept>`"
- Если вердикт = забраковано: "Все концепты забракованы. Рекомендуется запустить `/fnr-concept` заново с дополнительным контекстом."
