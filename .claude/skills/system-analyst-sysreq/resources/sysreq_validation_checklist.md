# Чек-лист валидации документа системных требований

## 1. Структура документа (шаблон Confluence)

- [ ] Документ начинается с заголовка `[СТ] <PROJECT>-<NNN> <Название>`
- [ ] **Раздел 1 — Введение:**
  - [ ] Таблица метаданных (наименование продукта, ответственные, Epic, БФТ, статус)
  - [ ] Термины и определения (таблица)
  - [ ] Ссылки (таблица)
  - [ ] История изменений (таблица: Дата | Автор | Суть изменений)
- [ ] **Раздел 2 — Общее описание:**
  - [ ] As-Is: описание текущего поведения с код-доказательствами
  - [ ] Ключевые компоненты (таблица: Компонент | Роль | Файл:строка)
  - [ ] Ограничения текущего решения (нумерованный список с доказательствами)
  - [ ] Ссылка на архитектурное решение
  - [ ] Диаграмма компонентов (PlantUML)
  - [ ] Схема последовательности (PlantUML)
- [ ] **Раздел 3 — План миграции:**
  - [ ] Диаграмма этапов (PlantUML activity diagram)
  - [ ] Таблица этапов (Этап | Действие | Затрагиваемые объекты | Откат)
  - [ ] Критерии готовности для каждого этапа
- [ ] **Раздел 4 — Функциональные требования (Backend / БД / API):**
  - [ ] Содержит только серверные задачи: БД, backend-логика, API, фоновые процессы, межсервисные интеграции
  - [ ] Каждый блок = логическая группа задач
  - [ ] Каждая задача содержит все обязательные подсекции (см. п.2)
- [ ] **Раздел 5 — Требования к интерфейсам (Frontend / UI):**
  - [ ] Содержит только клиентские задачи: верстка, UI-компоненты, клиентские API-интеграции, микроразметка, роутинг
  - [ ] Если UI-изменений нет — раздел опущен с пометкой «Не применимо»
  - [ ] Если есть — каждая задача содержит обязательные подсекции (см. п.2)
- [ ] **Разделение Backend / Frontend:**
  - [ ] Задачи корректно распределены между разделами 4 и 5 (нет серверных задач в разделе 5 и наоборот)
  - [ ] Смежные фичи (требуют и BE, и FE) представлены двумя задачами в разных разделах, связанными через «Зависимости»
  - [ ] [CRITICAL] Серверная задача помещена в раздел 5 или клиентская — в раздел 4 — 🔴
- [ ] **Ревью требований** — таблица согласования (аналитик, разработка, тестирование)
- [ ] **Риски и ограничения:**
  - [ ] Таблица рисков (ID | Риск | Вероятность | Влияние | Митигация)
  - [ ] Нумерованный список ограничений
- [ ] [CRITICAL] Пропущен любой из обязательных разделов — 🔴

## 2. Качество каждой задачи

- [ ] Таблица метаданных (Ответственный за тех. реализацию + Задача на разработку со статусом)
- [ ] Описание функционала — нумерованный список с подпунктами (a, b, c)
- [ ] Обоснование — почему выбран этот подход
- [ ] Затрагиваемые компоненты — таблица (Компонент | Тип изменения | Файл:строка)
- [ ] Критерии приёмки — нумерованный список проверяемых условий
- [ ] Зависимости — явно указаны (нет / после задачи N.M)
- [ ] [CRITICAL] Описание без обоснования и таблицы затрагиваемых компонентов — 🔴

## 3. API-требования (если применимо)

- [ ] Таблица эндпоинтов (Метод | Путь | Тип | Целевой топик/Описание)
- [ ] JSON-примеры формата ответа (в блоках кода `json`)
- [ ] Таблица статусов (Статус | Описание)
- [ ] HTTP-коды ответов указаны корректно

## 4. Обработка данных (если применимо)

- [ ] Таблица маршрутизации (Сущность | Метод вызова | Отправляемые события)
- [ ] Таблица источников данных (Сущность | DataSource | SQL-запрос)
- [ ] SQL-запросы в блоках кода `sql` с комментариями

## 5. Нефункциональные требования

- [ ] NFR — конкретные и измеримые (имена классов, размеры пулов, таймауты)
- [ ] Нумерованный список
- [ ] [CRITICAL] Нет оценочных формулировок («быстро», «надёжно», «удобно»)

## 6. План миграции

- [ ] Этапы упорядочены логически (зависимости учтены)
- [ ] Каждый этап содержит критерий готовности (как проверить)
- [ ] Каждый этап содержит процедуру отката
- [ ] Диаграмма миграции (PlantUML) соответствует текстовому плану

## 7. Риски и ограничения

- [ ] Каждый риск содержит оценку вероятности и влияния
- [ ] Каждый риск содержит конкретную митигацию
- [ ] Ограничения явно перечислены (что решение НЕ покрывает)
- [ ] Зафиксированы компоненты, которые НЕ должны быть затронуты (backward compatibility)

## 8. Визуализация

- [ ] Все диаграммы используют PlantUML (Mermaid запрещён)
- [ ] Диаграмма компонентов корректно отражает архитектуру
- [ ] Схема последовательности отражает порядок взаимодействия
- [ ] Все надписи на стрелках — на русском языке
- [ ] Технические имена указаны в скобках
- [ ] Все блоки PlantUML корректно закрыты (`@startuml` / `@enduml`)

## 9. Форматирование

- [ ] Нумерация разделов последовательная (1, 2, 3... / 3.1, 3.2... / 3.1.1, 3.1.2...)
- [ ] Таблицы используются для структурированных данных
- [ ] JSON/SQL — в блоках кода с соответствующей разметкой
- [ ] [CRITICAL] Нет абсолютных путей (`file:///`, `/Users/...`) — только относительные

## 10. Фактологическая точность

- [ ] Все утверждения в As-Is подтверждены код-доказательствами (файл:строка)
- [ ] Совпадают названия таблиц, полей, методов, классов с реальными в коде
- [ ] [CRITICAL] Утверждения, которые нельзя подтвердить кодом, помечены `[NEEDS_INVESTIGATION]`

## Hard Gates (Условия СТРОГОГО 🔴)

1. **Template Gate:** Отсутствует обязательный раздел (Введение, Общее описание с As-Is, План миграции, Функциональные блоки, Ревью, Риски и ограничения) — 🔴.
2. **As-Is Gate:** Раздел «Описание текущего поведения» без код-доказательств — 🔴.
3. **Task Gate:** Задачи без обоснования, таблицы затрагиваемых компонентов или критериев приёмки — 🔴.
4. **Acceptance Gate:** Критерии приёмки непроверяемые («работает корректно» без конкретики) — 🔴.
5. **NFR Gate:** NFR без измеримых критериев или с оценочными формулировками — 🔴.
6. **Migration Gate:** План миграции без процедур отката — 🔴.
7. **Risks Gate:** Риски без митигации — 🔴.
8. **PlantUML Gate:** Диаграммы в формате Mermaid или с незакрытыми блоками — 🔴.
9. **Path Gate:** Абсолютные пути в ссылках — 🔴.
10. **Hallucination Gate:** Утверждение о поведении системы, не подтверждённое кодом и не помеченное `[NEEDS_INVESTIGATION]` — 🔴.
