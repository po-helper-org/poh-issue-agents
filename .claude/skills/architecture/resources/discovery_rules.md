# 🗺 Универсальные правила обнаружения (Agnostic Discovery Protocol)

Если проект тебе незнаком, используй эти эвристические правила для быстрого поиска скрытых связей и границ системы.

## 1. Поиск «Внешних Миров» (Egress Discovery)
Ищи следы взаимодействия с внешней средой:
- **Сетевые клиенты:** Поиск по ключевым словам протоколов: `http://`, `https://`, `soap`, `grpc`, `amqp`, `kafka`. 
- **Библиотеки связи:** Ищи вызовы конструкторов сетевых клиентов (напр. `Fetch`, `Axios`, `HttpClient`, `Guzzle`, `Producer`).
- **Маппинг систем:** Если найден домен или IP — сопоставь его с бизнес-названием. (напр. `bank.api.com` -> `System_Ext(Bank, "Payment Gateway")`).

## 2. Поиск «Точек Хранения» (Data Stores Discovery)
- **Конфигурационный контекст:** Начни с поиска файлов с именами `config`, `env`, `params`, `settings`, `application`, `db`.
- **Строки подключения:** Ищи `DSN`, `Connection String`, `Host`, `Port`, `user:password`.
- **Топология БД:** Если в конфигах более одного уникального хоста БД — отрисовывай их как независимые `ContainerDb`.

## 3. Определение Границ Доменов (Domain Heheuristics)
- **Файловая группировка:** Если папка или пакет содержит более 5 файлов с общим префиксом/именем (напр. `Cart...`, `User...`, `Payment...`) — классифицируй это как самостоятельный Бизнес-контейнер (`Boundary/Domain`).
- **Связность данных:** Если набор классов оперирует одной и той же «Якорной сущностью» (напр. `OrderRecord`, `OrderSvc`, `OrderRepo`) — это один домен.

## 4. Маппинг «Публичных Имен» (Public-to-Internal Mapping)
- **Routing First:** Исследование любого проекта начинай с реестра путей (`Routes`, `Controllers mapping`, `Topics registry`, `CLI commands`).
- **Alias Resolution:** Если внешний эндпоинт (напр. `/api/profile-info`) ведет к методу с другим техническим именем (напр. `get_data`) — **ОБЯЗАТЕЛЬНО** заноси оба имени в словарь. Имя из внешнего интерфейса имеет приоритет для бизнес-документации.

## 5. Иерархия Приоритетов (Truth Priority)
1. **Код (Code):** Фактическая реализация метода.
2. **Конфиги (Config Context):** Физические адреса и настройки.
3. **Комментарии/Документация:** Только как вспомогательный материал.
