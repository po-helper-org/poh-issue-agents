# FNR-2: Автономная генерация решения по Issue командой `/analyze` (Слой C)

> **Тип:** Новая фича (архитектурное изменение — новый on-demand слой аналитики)
> **Дата:** 2026-07-22
> **Статус:** Открыта

---

## 1. Постановка проблемы

В сервисе нет пути исполнения, связывающего команду в комментарии Issue с пайплайном аналитики SA-helper: комментарий `/analyze` не распознаётся как команда, а стадия аналитики Слоя C (`run_research_pipeline`) — заглушка `NotImplementedError`. Как следствие, пользователь не может получить автономно спроектированное решение («как реализовать эту задачу») прямо в Issue.

## 2. Контекст

Целевой сценарий (HowToDemo): в открытом Issue пользователь пишет `/analyze`; Обработчик Issue реагирует, что взял команду в работу; следующим сообщением автономно (без вмешательства человека) отдаёт аналитические артефакты, спроектированные через SA-helper (FNR-пайплайн: постановка → концепт → дебаты → системные требования → валидация). Пример целевого Issue — `po-helper-org/poh-pr-agents#5` (ревизия слоя reliability, задача ссылается на конкретные файлы `queue.py`, `gateway.py` и т.д., то есть требует код-обоснованного анализа целевого репозитория).

Разрыв обнаружен при попытке пройти сценарий на текущей кодовой базе: событие `issue_comment.created` доходит до воркфлоу, но нигде не превращается в запуск SA-helper. Функция-мишень для этой логики (`run_research_pipeline`) существует только как заглушка. Проблема относится к запланированному, но не реализованному «Слою C — аналитика по запросу» (`docs/roadmap-post-layer-a.md:31-54`).

## 3. Текущее поведение (As-Is)

Приём комментариев работает как чистый транспорт: `webhook` проверяет подпись, отсекает комментарии от ботов и **безусловно** сигналит `user_comment` в Обработчик Issue, не заглядывая в тело комментария — диспатча команд нет. Обработчик Issue кладёт текст в очередь сигналов с префиксом `__comment__:`, и этот сигнал потребляется **только** циклом уточнений приёмного шлюза (пока `gate.status == "VAGUE"`) либо точками ожидания решения человека, где строка, не совпавшая с лейблом, приводит к завершению воркфлоу. Даже если бы путь существовал, тяжёлая стадия аналитики — заглушка. Исполнительная среда для `claude -p` (skills, commands, repomix) в образ воркера не уложена.

**Цепочка событий (от симптома к корню):**

| Шаг | Компонент | Что происходит | Доказательство |
|-----|-----------|---------------|----------------|
| 1 | Webhook | Приходит `issue_comment.created`; после guard'а на `Bot` тело комментария **не инспектируется** на команды | `webhook/main.py:89-96` |
| 2 | Webhook | Безусловно вызывается `handle.signal("user_comment", body)` — любой комментарий трактуется одинаково | `webhook/main.py:103` |
| 3 | Обработчик Issue | `user_comment` кладёт `__comment__:<text>` в очередь сигналов | `worker/workflows.py:42-43` |
| 4 | Обработчик Issue | Сигнал читается только в цикле уточнений (`while gate.status == "VAGUE"`) | `worker/workflows.py:107-116` |
| 5 | Обработчик Issue | Вне цикла воркфлоу либо уже завершён (advisor-ответ), либо ждёт лейбл в `_wait_for_signal()`; строка-комментарий там не совпадает с `research-me`/`bug-me` и уходит в `else: return` | `worker/workflows.py:174`, `worker/workflows.py:190-191` |
| 6 | Research-пайплайн | Целевая функция тяжёлой аналитики — заглушка: `raise NotImplementedError` | `worker/activities.py:280-287` |
| 7 | Образ воркера | В контейнер копируются только `worker/` и `shared/`; `.claude/skills` и `.claude/commands` отсутствуют, `repomix` не установлен | `worker/Dockerfile:21-22`, `docker-compose.yml:72-75` |

**Ключевые компоненты:**

| Компонент | Роль в контексте проблемы | Файл:строка |
|-----------|------|-------------|

## 4. Корень проблемы

Корень — **структурный, состоит из трёх независимых разрывов**, каждый из которых сам по себе блокирует сценарий:

1. **Транспорт не диспатчит команды.** `webhook` обрабатывает `issue_comment.created` единственным способом — сигналом `user_comment`, без разбора тела на команды. **Доказательство:** `webhook/main.py:103` — единственный вызов на ветке комментариев, ему предшествует только guard на `Bot` (`:95-96`), тело не читается как команда.

2. **Потребитель комментария имеет узкую семантику.** Даже дойдя до воркфлоу, `__comment__:` читается лишь в цикле уточнений; после триажа воркфлоу либо завершён, либо припаркован на ожидании лейбла, где произвольная строка ведёт к `return`. **Доказательство:** `worker/workflows.py:107-116` (единственное чтение внутри `while VAGUE`) и `worker/workflows.py:190-191` (`else: return` на несовпавшем сигнале в точке решения человека).

3. **Целевая стадия аналитики не реализована, а её исполнительная среда не собрана.** **Доказательство:** `worker/activities.py:287` — `raise NotImplementedError`; `worker/Dockerfile:21-22` — в образ копируются только `worker/`+`shared/` (нет `.claude/skills`, `.claude/commands`, нет `repomix`); `worker/llm.py:28-30` фиксирует, что z.ai отвергает OpenAI tool-calling — работоспособность tool-calling `claude -p` через Anthropic-эндпоинт z.ai (`.env.example:20`) в проекте ещё не подтверждена.

Симптом («`/analyze` ничего не делает») — следствие. Причина — отсутствие сквозного пути «команда в комментарии → диспетчер → SA-helper пайплайн → артефакты в Issue» и неукомплектованность среды его исполнения.

## 5. Ожидаемое поведение

1. Комментарий, начинающийся с `/analyze`, распознаётся как команда независимо от текущего состояния триажа Issue (воркфлоу активен, завершён или припаркован).
2. Обработчик Issue немедленно и видимо подтверждает приём команды (реакция и/или короткий комментарий «взял в работу»).
3. Автономно, без вмешательства человека, отрабатывает FNR-пайплайн SA-helper, код-обоснованный на кодовой базе целевого репозитория (постановка → концепт → дебаты → системные требования → валидация).
4. Полученные артефакты доставляются в Issue: сохраняются в ветку `research/issue-<n>`, а итоговый комментарий содержит ответ «как реализовать» со ссылками на артефакты.
5. Повторный `/analyze` по тому же Issue не порождает дублирующий прогон (идемпотентность).

## 6. Зона воздействия

**Прямое воздействие:**

| Компонент | Тип | Файл |
|-----------|-----|------|
| Webhook | Точка входа (диспатч команд) | `webhook/main.py` |
| Обработчик Issue | Temporal-workflow (диспатч команды и запуск стадии аналитики) | `worker/workflows.py` |
| Стадии аналитики | Temporal-activities (реализация вместо заглушки) | `worker/activities.py` |
| Клиент GitHub | REST-обёртка (реакция на комментарий, пуш артефактов в ветку) | `worker/github_client.py` |
| Регистрация воркера | Список workflows/activities | `worker/worker.py` |
| Типы воркфлоу | Датаклассы входных данных | `shared/workflow_types.py` |
| Образ воркера | Сборка (skills/commands/repomix, env для `claude -p`) | `worker/Dockerfile` |
| Конфиг окружения | Переменные бэкенда skills | `.env.example` |
| Оркестрация | Монтирование `.claude`, рабочая директория | `docker-compose.yml` |

**Косвенное воздействие (зависимые компоненты):**

| Компонент | Как зависит | Что может сломаться |
|-----------|------------|------|
| `duplicate_check` | Ищет ветки `research/issue-<n>`/`bug/issue-<n>` для reuse-note | При смене имени ветки артефактов reuse-note перестанет находить наработки | 
| Цикл уточнений приёмного шлюза | Делит очередь сигналов `user_comment` с новым диспатчем команд | Команда `/analyze`, поданная во время ожидания уточнения, может быть съедена как ответ на вопрос | 
| Точки решения человека | `_wait_for_signal()` ждёт лейблы; новый сигнал команды меняет набор ожидаемых значений | Ошибочный `return` при добавлении новой ветки сигналов |
| Детектор дедлоков / таймауты | Тяжёлый прогон `claude -p` — длинная activity | Ложные срабатывания TMPRL1101 / task timeout (уже прецедент в `worker/worker.py:44-54`) |

**Защищённые компоненты (не должны быть затронуты):**

- Логика триажа Слоя A — `intake_gate`, `classify_issue`, `duplicate_check`, `score_priority` (`worker/activities.py:105-274`): автономный триаж должен продолжать работать без изменений поведения.
- Механизм `DRY_RUN` (`worker/github_client.py:19`, `:62-63`): новые мутирующие вызовы обязаны его соблюдать.
- Инвариант «single-node, zero external infra» (`docs/roadmap-post-layer-a.md`, issue #1 «Открытых решений»): решение не должно неявно вводить новые внешние зависимости инфраструктуры.
- Идемпотентность по `workflow_id` (`webhook/main.py:44-45`, `worker/workflows.py:1-16`): повторные события не должны плодить сущности.
- Сами skills SA-helper (`.claude/skills`, `.claude/commands`): по договорённости конфига меняется только backend через `ANTHROPIC_*`, скиллы не редактируются (`.env.example:18-19`).

## 7. Ограничения

- Скиллы SA-helper и FNR-команды не модифицируются — используются как есть; допускается только смена backend-модели через `ANTHROPIC_BASE_URL`/`ANTHROPIC_AUTH_TOKEN` (`.env.example:18-21`).
- Бэкенд LLM — z.ai; для `claude -p` используется Anthropic-совместимый эндпоинт (`.env.example:20`). Корректность tool-calling (`Read`/`Write`/`Bash`) на этом эндпоинте в проекте не верифицирована (`worker/llm.py:28-30` фиксирует несовместимость OpenAI tool-calling у GLM) — это внешнее для задачи техническое ограничение, влияющее на осуществимость.
- Сохраняется одноузловая zero-infra модель развёртывания (Temporal + Postgres + worker + webhook в одном `docker-compose`, `docker-compose.yml`), без обязательного добавления Redis/брокеров ради этой фичи.
- Артефакты FNR код-обоснованы: каждой стадии нужен доступ к коду целевого репозитория, а не только к тексту Issue (требование самих FNR-команд — «каждое утверждение подтверди код-доказательством `файл:строка`»).
- Новые GitHub-мутации (реакции, ветки, коммиты) обязаны соблюдать `DRY_RUN` и работать в рамках прав GitHub App: Issues (r/w), Pull requests (r/w), Contents (r/w) (`.env.example:2-3`).
