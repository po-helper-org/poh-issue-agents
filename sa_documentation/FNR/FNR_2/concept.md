# Концепты решений — FNR-2

> **Задача:** Автономная генерация решения по Issue командой `/analyze` (Слой C)
> **Дата:** 2026-07-22
> **Статус:** Концепты сгенерированы (ожидает `/fnr-debate`)

---

## Резюме проблемы

Нет сквозного пути «команда `/analyze` в комментарии → диспетчер → пайплайн SA-helper → артефакты в Issue». Разрыв тройной: транспорт не диспатчит команды (`webhook/main.py:103`), потребитель `user_comment` имеет узкую семантику цикла уточнений (`worker/workflows.py:107-116`), а целевая стадия аналитики — заглушка (`worker/activities.py:287`) с неукомплектованной средой исполнения `claude -p` (`worker/Dockerfile:21-22`).

## Зафиксированные параметры решения (форки заказчика — общие для всех концептов)

Эти оси уже выбраны и не пересматриваются концептами; концепты различаются **моделью исполнения и триггером**, а не этими параметрами:

- **Глубина пайплайна:** полная цепочка FNR — `task → concept → debate → system_requirements → validate`.
- **Код-обоснование:** клонирование целевого репозитория + `repomix` (упаковка один раз, переиспользуется всеми стадиями).
- **Доставка:** артефакты коммитятся в ветку `research/issue-<n>` + итоговый комментарий-сводка со ссылками.
- **Триггер:** гибрид — сигналить активный `IssueLifecycle`, иначе стартовать отдельный воркфлоу аналитики.

## Концепт A: Выделенный воркфлоу аналитики + headless SA-helper (постадийные activities) [Категория: Правильное]

**Суть:** Ввести отдельный Temporal-workflow `IssueAnalysis` (фикс. id `analysis-<repo>-<n>`), исполняющий FNR-цепочку как **пять отдельных activities** (по одной на стадию `claude -p`), каждая со своим таймаутом/ретраем. Диспатч `/analyze` в webhook; гибридная маршрутизация к воркфлоу. Клонирование+`repomix`+коммит артефактов — отдельными activities вокруг стадий.

**Что менять:**

| Компонент | Изменение | Файл |
|-----------|----------|------|
| Webhook | Разбор тела комментария на `/analyze`; describe-хэндла и маршрутизация signal-или-start | `webhook/main.py:89-107` |
| Типы | Датакласс `AnalyzeInput(repo, issue_number, comment_id, …)` | `shared/workflow_types.py:4-12` |
| Воркфлоу | Новый класс `IssueAnalysis` + сигнал `analyze_requested` на `IssueLifecycle`, спавнящий дочерний воркфлоу с `ParentClosePolicy.ABANDON` | `worker/workflows.py:31-43` |
| Activities | `prepare_workspace` (clone+repomix), `run_fnr_stage(stage)` ×5, `publish_artifacts` (branch+comment), `ack_command` | `worker/activities.py:280-287` (замена заглушки) |
| Клиент GitHub | `add_reaction(comment_id)`, `push_artifacts_to_branch(...)` | `worker/github_client.py:61-109` |
| Регистрация | Зарегистрировать `IssueAnalysis` и новые activities | `worker/worker.py:26-41` |
| Образ | Уложить `.claude/skills`+`.claude/commands` в `~/.claude`; установить `repomix`; env `ANTHROPIC_*` для subprocess | `worker/Dockerfile:3-22` |
| Оркестрация | Смонтировать `.claude` (или запечь в образ), рабочая директория под клоны | `docker-compose.yml:72-75` |

**Плюсы:**
- Максимальная наблюдаемость: каждая стадия — отдельное событие в Temporal UI, ретраится точечно.
- Чистая изоляция: on-demand аналитика не связана с состоянием триажа.
- Гранулярный откат: сбой стадии 4 не перезапускает клон+repomix.

**Минусы:**
- Постадийные activities делят рабочую директорию на локальном диске — при нескольких воркерах-репликах файлы стадии могут не дожить до следующей activity (на одном узле не проблема, но это неявная привязка к single-node).
- Наибольший объём плумбинга (5 activities + оркестрация состояния воркспейса между ними).

**Оценка усилий:** 2 недели (новый воркфлоу + 5 activities + образ + github_client + спайк tool-calling).

**Риски:**

| Риск | Вероятность | Митигация |
|------|------------|-----------|
| Потеря файлов воркспейса между activities на репликах | Средняя | Привязать воркфлоу к одному воркеру (task queue) либо передавать артефакты через объектное хранилище |
| tool-calling `claude -p` через Anthropic-эндпоинт z.ai не работает | Высокая | Спайк №1 до любой другой работы (`worker/llm.py:28-30`) |
| Суммарное время 5 стадий > бюджет | Средняя | Отдельные таймауты на стадию, shallow-clone |

**Откат:** Удалить регистрацию `IssueAnalysis`/новых activities (`worker/worker.py`), вернуть заглушку; webhook-диспатч за флагом — снять флаг. Существующий триаж не затронут.

## Концепт B: Гибридный триггер + пайплайн одной heartbeat-activity [Категория: Прагматичное]

**Суть:** Тот же пайплайн и та же доставка, но вся цепочка «clone → repomix → 5×`claude -p` → collect → commit → comment» — **одна оркестрирующая activity** `run_analysis_pipeline` с heartbeat между стадиями. Триггер — гибрид: `webhook` сигналит активный `IssueLifecycle` (`analyze_requested`), иначе стартует `IssueAnalysis`; оба сходятся на воркфлоу с фикс. id `analysis-<repo>-<n>` (идемпотентность даёт Temporal через `WorkflowAlreadyStarted`).

**Что менять:**

| Компонент | Изменение | Файл |
|-----------|----------|------|
| Webhook | Разбор `/analyze`; `try signal(analyze_requested) except → start_workflow(IssueAnalysis)` | `webhook/main.py:89-107` |
| Воркфлоу | `IssueAnalysis` (одна activity внутри) + сигнал `analyze_requested` на `IssueLifecycle`, спавнящий дочерний воркфлоу `ABANDON` | `worker/workflows.py:31-43` |
| Activities | Одна `run_analysis_pipeline(issue, comment_id)` — весь жизненный цикл воркспейса внутри, heartbeat на каждой стадии; `retry=1`, `start_to_close≈75м`, `heartbeat_timeout≈5м` | `worker/activities.py:280-287` |
| Клиент GitHub | `add_reaction`, `push_artifacts_to_branch` | `worker/github_client.py:61-109` |
| Типы / Регистрация / Образ / Оркестрация | Как в A (общая часть) | `shared/workflow_types.py`, `worker/worker.py`, `worker/Dockerfile`, `docker-compose.yml` |

**Плюсы:**
- Файловая консистентность бесплатно: единый жизненный цикл воркспейса в одной activity, нет проблемы «файлы между activities».
- Меньше плумбинга, чем в A; heartbeat закрывает уже известную боль таймаутов/детектора дедлоков (`worker/worker.py:44-54`).
- Гибрид переиспользует живой воркфлоу, когда он есть; фикс. id гарантирует один прогон на Issue.

**Минусы:**
- Грубая гранулярность ретрая: сбой на стадии 5 перезапускает с клона (для `retry=1` это в любом случае не авторетраится — цена мала).
- Гибрид добавляет два пути входа и сигнал-хендлер, спавнящий дочерний воркфлоу; при этом FNR-пайплайн не переиспользует контекст триажа (выигрыш гибрида — только «не плодить воркфлоу», а не экономия контекста).

**Оценка усилий:** 1 неделя (одна activity + гибрид-обвязка + образ + github_client + спайк).

**Риски:**

| Риск | Вероятность | Митигация |
|------|------------|-----------|
| tool-calling `claude -p` через z.ai не работает | Высокая | Спайк №1 до реализации (общий с A) |
| Гонка describe→signal (воркфлоу завершился между проверкой и сигналом) | Средняя | Фолбэк: на ошибке signal — `start_workflow(IssueAnalysis)`; фикс. id снимает дубли |
| Длинная activity упирается в таймаут | Средняя | Heartbeat + запас по `start_to_close`; shallow-clone |

**Откат:** Снять флаг диспатча в webhook и вернуть заглушку activity; сигнал `analyze_requested` — no-op. Триаж не затронут.

## Концепт C: Переиспользовать лейбл-путь + один subprocess-блоб, без нового воркфлоу [Категория: Быстрый костыль]

**Суть:** `/analyze` в webhook просто вешает лейбл `research-me` (или напрямую слёт `human_decision`), а `run_research_pipeline` реализуется как один subprocess-блоб `claude -p`, печатающий один markdown-документ, который постится **одним комментарием** (с усечением до лимита GitHub). Ни нового воркфлоу, ни ветки, ни постадийности.

**Что менять:**

| Компонент | Изменение | Файл |
|-----------|----------|------|
| Webhook | На `/analyze` — `handle.signal("human_decision", "research-me")` | `webhook/main.py:83-87` |
| Activity | Заполнить `run_research_pipeline` одним subprocess-вызовом, вернуть текст в комментарий | `worker/activities.py:280-287` |
| Образ | Уложить skills/commands, установить repomix | `worker/Dockerfile:21-22` |

**Плюсы:**
- Минимум кода; переиспользует существующую точку решения человека (`worker/workflows.py:176-182`).
- Быстрый путь к демо.

**Минусы:**
- `/analyze` работает только если воркфлоу жив и находится ровно в точке ожидания `research-me`, и только для `advisor:feature-request` (`worker/workflows.py:176`) — на закрытом/завершённом Issue не сработает.
- Один комментарий не вмещает 5 артефактов (лимит 65536, PlantUML сырьём); теряется код-обоснование (нет ветки с файлами).
- Нет идемпотентности, нет наблюдаемости по стадиям, лейбл-семантика перегружается.

**Оценка усилий:** 2-3 дня.

**Риски:**

| Риск | Вероятность | Митигация |
|------|------------|-----------|
| Воркфлоу не в нужном состоянии → команда молча теряется | Высокая | Нет хорошей митигации в рамках концепта (это его врождённый минус) |
| Обрезка документа теряет суть sysreq | Высокая | Постить только concept-сводку, но тогда цель «системные требования» не достигнута |

**Откат:** Убрать ветку `/analyze`→label в webhook; вернуть заглушку.

## Концепт D: Обойти headless-агента — sysreq одним structured-вызовом через рабочий Instructor-путь [Категория: Креативный хак]

**Суть:** Не запускать `claude -p`/skills/repomix вообще. Собрать один большой системный промпт (переложив суть FNR-шаблонов) и вызвать существующий `llm.extract(...)` (z.ai OpenAI/JSON-mode, который **уже доказанно работает** — `worker/llm.py:41-53`), выдав документ, приближающий sysreq, за один-два structured-вызова. Код-контекст — через `gh`/git-grep, вложенный в промпт. Артефакты — в ветку одним файлом.

**Что менять:**

| Компонент | Изменение | Файл |
|-----------|----------|------|
| Webhook | Диспатч `/analyze` (как в B, минимально) | `webhook/main.py:89-107` |
| Activity | `run_analysis_pipeline` поверх `llm.extract` с Pydantic-схемой «sysreq», без subprocess | `worker/activities.py:280-287`, `worker/llm.py:41-53` |
| Клиент GitHub | `push_artifacts_to_branch` (один файл) + comment | `worker/github_client.py:61-109` |

**Плюсы:**
- **Полностью снимает риск №1** (не зависит от tool-calling `claude -p` через z.ai) — использует уже рабочий путь.
- Нет изменений образа под skills/repomix; быстрее и дешевле по времени прогона.

**Минусы:**
- Теряется настоящая FNR-фидельность: нет отдельных ролей (problem-analyst/solution-designer/…), нет дебатов, слабее код-обоснование `файл:строка` (модель не ходит по файлам агентно, а видит вкладку контекста).
- Расходится с целью «спроектировать через sa-helper»: это имитация артефактов, а не прогон скиллов.
- Ограничение контекстного окна на большом репозитории.

**Оценка усилий:** 3-4 дня.

**Риски:**

| Риск | Вероятность | Митигация |
|------|------------|-----------|
| Качество/галлюцинации без агентного хождения по коду | Средняя | Вкладывать точечные выдержки кода (gh search) в промпт |
| Заказчик считает это «не через sa-helper» | Высокая | Позиционировать как fallback, если спайк №1 провалится |

**Откат:** Тривиальный — одна activity + один диспатч.

## Концепт E: «Не трогай» — оставить ручной `research-me` до прохождения спайка *(условный)*

**Обоснование:** Если спайк tool-calling `claude -p` через Anthropic-эндпоинт z.ai проваливается (риск помечен как высокий — `worker/llm.py:28-30`), автономный headless-прогон недостижим без смены бэкенда. Тогда полноценный `/analyze` даёт ложное ощущение автономии при непредсказуемых сбоях. Разумнее оставить ручной запуск SA-helper + существующую точку `research-me`, пока не выбран рабочий бэкенд. Уместно **только** как исход при провале спайка, не как основной путь.

## Дебаты (Архитектор vs Адвокат Дьявола)

### Раунд 1: Защита концепта B (Гибрид + одна heartbeat-activity)

**Архитектор:**
- Одна activity даёт файловую консистентность бесплатно: клон, `repomix`-упаковка и 5 стадий пишут в общий воркспейс на локальном диске одного процесса. В A постадийные activities делят каталог, а воркер сконфигурирован на `max_concurrent_activities=8` (`worker/worker.py:54`) — то есть activities разных Issue уже идут параллельно; полагаться на переживание файлов стадии между отдельными activity-тасками — неявная привязка к single-node, которую B устраняет.
- Heartbeat напрямую бьёт по уже известной боли: команда специально включила `debug_mode=True` из-за ложных срабатываний детектора дедлоков TMPRL1101 под долгими стадиями (`worker/worker.py:44-51`). Длинный прогон `claude -p` — ровно этот класс; heartbeat + запас `start_to_close` держат таск живым без хаков.
- Гибрид переиспользует существующую машинерию сигналов: `IssueLifecycle` уже принимает сигналы (`worker/workflows.py:37-43`), добавляется один `analyze_requested`; фикс. id `analysis-<repo>-<n>` даёт идемпотентность через штатный `WorkflowAlreadyStarted`, как это уже сделано для Issue-воркфлоу (`webhook/main.py:44-45`, `worker/workflows.py:1-16`).
- Доставка в ветку переиспользует существующее соглашение: `duplicate_check` уже ищет `research/issue-<n>` для reuse-note (`worker/activities.py:199-204`), поэтому артефакты в этой же ветке — консистентны, а не вводят новую сущность.
- Меньше плумбинга, чем A (одна activity против пяти + оркестрация воркспейса), при той же наблюдаемости на уровне комментариев-статусов.

**Адвокат Дьявола:**
- А что если воркер перезапустится в середине 75-минутной activity? `docker-compose.yml:71` — `restart: unless-stopped`; при рестарте activity-таск переедет, но `retry=1` (концепт B) означает отсутствие авторетрая — весь прогон схлопнется в FAILED после десятков минут работы. В A погибла бы только текущая стадия.
- Гибрид — это два пути входа, и оба должны сойтись корректно. Сигнал-хендлер `analyze_requested`, спавнящий дочерний воркфлоу с `ABANDON` из обработчика сигнала, выполняется, пока `run()` припаркован в `_wait_for_signal()` (`worker/workflows.py:174`) — нужно доказать, что спавн ребёнка из хендлера сигнала не конфликтует с основным циклом; иначе будет тихая гонка.
- Коллизия очереди сигналов: сегодня webhook шлёт `user_comment` на все комментарии (`webhook/main.py:103`), а цикл уточнений читает их (`worker/workflows.py:107-116`). Если `/analyze` придёт, пока Issue в статусе VAGUE, кто его получит — диспетчер команд или цикл уточнения? При неверном разведении команда либо съедается как ответ на уточняющий вопрос, либо дублируется.
- «Гибрид переиспользует живой воркфлоу» — а что он с этого имеет? FNR-пайплайн клонирует репозиторий и обосновывается на коде, ему не нужны `classification`/`dup`/`priority` из памяти триажа. Значит единственная ценность сигналить живой воркфлоу — «не плодить сущность», которую фикс. id и так закрывает. Сложность гибрида не окупается.
- Риск №1 (tool-calling `claude -p` через Anthropic-эндпоинт z.ai) в B никак не снят — а он помечен «Высокая». Если он не работает, B нереализуем целиком, и мы это узнаем поздно.

### Раунд 2: Пересмотр

**Архитектор:**
- По рестарту воркера: соглашаюсь, это реальный edge. Митигация без ухода в A — (а) heartbeat уже делает прогресс видимым; (б) на терминальный сбой activity вешаем `advisor:error`-подобный коммент (переиспользуя паттерн `post_error_label`, `worker/activities.py:137-142`), чтобы пользователь мог повторить `/analyze` — фикс. id даёт чистый рестарт. On-demand-прогон по команде — best-effort по своей природе; потеря прогона не рушит данные, в отличие от триажа. Поднимать `max_attempts>1` не будем: `claude -p` недетерминирован, слепой авторетрай дорог (это же зафиксировано в `worker/workflows.py:181` для research-пайплайна — `maximum_attempts=1`).
- По спавну из хендлера сигнала: снимаю сложность — не спавню ребёнка из обработчика. Хендлер лишь ставит флаг/кладёт сентинел в очередь, а решение принимает webhook: `try: signal(analyze_requested) except → start_workflow(IssueAnalysis)`. То есть основной путь — прямой `start_workflow` дочернего воркфлоу из webhook по фикс. id; сигнал живому воркфлоу оставляем только как уведомление, не как исполнитель. Это упрощает гибрид до «диспатч в webhook», а тяжёлую работу всегда несёт выделенный `IssueAnalysis`.
- По коллизии сигналов: развожу явно в webhook — тело, начинающееся с `/analyze`, идёт по ветке команды и НЕ шлётся как `user_comment`; всё прочее — как сегодня (`webhook/main.py:103`). Цикл уточнений не увидит `/analyze`, диспетчер не увидит обычные ответы. Одна точка ветвления, тестируется юнит-тестом парсера.
- По «зачем гибрид»: принимаю аргумент частично — ценность гибрида свелась к диспатчу + фикс. id. Фактически концепт B сходится к «B, где исполнитель — всегда выделенный `IssueAnalysis`, а живому воркфлоу шлётся необязательное уведомление». Это ближе к A по чистоте, но сохраняет одну heartbeat-activity вместо пяти.
- По риску №1: делаю его gate. Первая задача реализации — спайк: один `claude -p --print`, пишущий файл, против `ANTHROPIC_BASE_URL=https://api.z.ai/api/anthropic` (`.env.example:20`). Пока спайк не зелёный — код B не пишем. Если спайк красный — пивот на D (обход через рабочий Instructor-путь `worker/llm.py:41-53`), который не зависит от tool-calling.

**Адвокат Дьявола:**
- Усиление принимаю: диспатч-в-webhook + всегда-выделенный `IssueAnalysis` снимает и гонку хендлера, и коллизию очереди. Остаётся вопрос эволюции: через 6-12 месяцев, когда воркеров станет >1 (горизонтальное масштабирование ради пропускной способности), одна 75-мин activity на общий локальный воркспейс всё ещё привязана к тому же узлу. Это осознанный долг?
- И ещё: `possible-duplicate` guard из роадмапа (`docs/roadmap-post-layer-a.md:60`) — тяжёлые стадии не должны стартовать без `confirmed-not-duplicate`. `/analyze` его игнорирует. Это намеренно?

**Архитектор:**
- Да, осознанный долг и он локализован: воркспейс инкапсулирован в одной activity, поэтому переезд на объектное хранилище/общий том — это изменение одной функции `run_analysis_pipeline`, не переписывание воркфлоу. Привязку к узлу для этой activity можно позже выразить отдельной task-queue. Для пилота single-node — зафиксированный инвариант (`task.md`, раздел 7).
- Про `possible-duplicate`: `/analyze` — явная команда человека, она сознательно оверрайдит эвристику дублей (человек уже посмотрел Issue). Оставляем как есть, но фиксируем в sysreq как явное проектное решение, чтобы не выглядело недосмотром.

### Раунд 3: Финальная позиция

**Архитектор:** Концепт B в усиленной форме — диспатч команды в webhook, исполнитель всегда выделенный `IssueAnalysis` (фикс. id `analysis-<repo>-<n>`), пайплайн одной heartbeat-activity, спайк tool-calling как gate, D как фолбэк — даёт нужную автономность при минимальном плумбинге и файловой консистентности, не ломая ни триаж, ни single-node-инвариант.

**Адвокат Дьявола:** Согласен при условии, что спайк №1 зелёный до реализации, коллизия очереди сигналов разведена одной точкой в webhook, а осознанные долги (single-node воркспейс, оверрайд `possible-duplicate`) явно зафиксированы в системных требованиях, а не растворены в коде.

## ВЕРДИКТ

**Выбранный концепт:** B (Гибрид + одна heartbeat-activity), принят **с модификациями**, вобрав чистоту A и хедж D.

**Модификации:**
1. **Спайк-gate:** первая задача — верификация tool-calling `claude -p` через Anthropic-эндпоинт z.ai (`.env.example:20`, риск из `worker/llm.py:28-30`); при провале — пивот на концепт D (обход через `llm.extract`, `worker/llm.py:41-53`).
2. **Исполнитель всегда выделенный:** тяжёлую работу несёт воркфлоу `IssueAnalysis` (фикс. id `analysis-<repo>-<n>`); webhook делает `try signal(analyze_requested) → except start_workflow`; спавн из хендлера сигнала исключён (снятие гонки).
3. **Разведение сигналов одной точкой:** в webhook тело `/analyze` → ветка команды и не отправляется как `user_comment` (`webhook/main.py:103`); цикл уточнений (`worker/workflows.py:107-116`) не затрагивается.
4. **Отказоустойчивость прогона:** heartbeat между стадиями; `retry=1`; на терминальный сбой — коммент об ошибке (паттерн `post_error_label`, `worker/activities.py:137-142`) + возможность повторить `/analyze` (чистый рестарт по фикс. id).
5. **Явно зафиксировать осознанные долги:** single-node воркспейс (инкапсулирован в `run_analysis_pipeline`, будущий переезд — одна функция) и намеренный оверрайд `possible-duplicate`-guard командой человека.

**Обоснование:** Адвокат не выявил неприемлемых рисков после усиления; ключевой внешний риск (tool-calling z.ai) вынесен в gate со спланированным фолбэком D, поэтому концепт реализуем инкрементально и обратимо, не нарушая защищённые компоненты триажа и single-node-инвариант.
