# FNR-3: Концепты решения — кластеризация под поставку

> **Вход:** `sa_documentation/FNR/FNR_3/task.md`
> **Изучение:** `docs/consolidation-clustering-study.md` (обзор практик + эксперимент)
> **Дата:** 2026-07-15
> **repomix:** отсутствует — концепты на основе живого кода (`worker/consolidation_*.py`).

Задача: заменить кластеризацию по `proposed_mechanism` (вырождается в одиночки)
на группировку по **оси поставки** (кластер = что реализуется/релизится вместе).
Инварианты: #111 target-guard, множественное членство, anchors, PR-only (не мутировать Issue).

---

## Концепт A — Taxonomy-first, 3 стадии (Правильное) ⭐ рекомендуется

**Суть.** Заменить `cluster_profiles` на три activity:
1. `derive_taxonomy(profiles)` — один LLM-вызов на весь бэклог → 8–12 **зон
   поставки** (name, boundary «что закрывает одна итерация», surface «движок/модуль»).
2. `assign_zone(profile, taxonomy)` — fan-out пер-Issue, классификация в primary-зону
   (+ secondary для сквозных).
3. `slice_zone(zone, members, profiles)` — крупную зону режет на инкременты
   (MVP/MVP+1) по зависимостям + потолку размера.

**Меняемые компоненты:**
- `worker/consolidation_activities.py` — убрать `cluster_profiles`/`_cluster_call`/`_merge_local`; добавить 3 activity + схемы `DeliveryZone`/`ZoneAssignment`/`Increment`.
- `worker/consolidation_workflow.py` — reduce-стадия: `derive_taxonomy` → fan-out `assign_zone` → (опц.) `slice_zone`.
- `prompts/` — `system_derive_taxonomy.md`, `system_assign_zone.md`, `system_slice_zone.md` (заменяют `system_cluster*.md`).
- `synthesize_unifying_issue`/`write_consolidation_pr` — вход «зона/инкремент» вместо `Cluster` (минимальная правка полей).
- `shared/workflow_types.py` — новые dataclass'ы.

**Плюсы.** Проверено экспериментом (8 зон на 70 Issue, инкременты, synth). Классификация в фикс-словарь надёжна. Чинит и качество, и replay-масштаб (стадия assign — компактные ярлыки). Ложится на #97/#133.
**Минусы.** Больше кода, чем костыль; нужны 3 новых промпта + тесты.
**Усилия.** ~2–3 дня (TDD, как FNR-2, ~8 задач).
**Риски.** Низкие — подход эмпирически подтверждён; главный открытый вопрос — качество `derive_taxonomy` на растущем бэклоге (митигация: сильная модель + few-shot).

## Концепт B — Two-stage без нарезки (Прагматичное)

**Суть.** Только стадии 1+2 (`derive_taxonomy` + `assign_zone`); зона = кластер
напрямую, `slice_zone` отложить. Крупная зона идёт как один объединяющий Issue
целиком (нарезку делает человек/следующая итерация).

**Меняемые компоненты.** Как A, минус `slice_zone` + его промпт/тесты.
**Плюсы.** Быстрее до рабочего результата; тот же скачок качества (8 зон vs одиночки); минимум нового.
**Минусы.** Зона в 14 Issue ≠ один релиз — «максимум за итерацию» не гарантирован без нарезки; slice всё равно понадобится.
**Усилия.** ~1–1.5 дня (~5 задач).
**Риски.** Низкие. Технический долг = отложенный slice.

## Концепт C — Коарс-механизм в промпте (Быстрый костыль)

**Суть.** Не трогать структуру: в `extract_solution_profile` заставить модель
писать `proposed_mechanism` из **фикс-энума грубых доменов** (`jira`, `memory-core`,
`bft`, `ingestion`, `sprint`, `llm-perf`, `ui`, `devops`), а `cluster_profiles`
группировать по совпадению этого поля (можно вообще без LLM — `group by`).

**Меняемые компоненты.** `prompts/system_solution_profile.md` (энум), `cluster_profiles` (группировка по полю).
**Плюсы.** Минимум кода (~2–4 ч); детерминированно; убирает открытую кластеризацию.
**Минусы.** Энум зашит вручную — не адаптируется к новым темам; нет boundary/surface/нарезки; грубее taxonomy-first; по сути ручная таксономия без вывода.
**Усилия.** ~2–4 часа.
**Риски.** Средние — жёсткий энум устаревает с бэклогом; качество ниже A/B. Годится как временная затычка, не как целевое.

## Концепт D — Эмбеддинги + HDBSCAN (Креативный хак)

**Суть.** Векторизовать текст Issue (эмбеддинг-эндпоинт / локальная модель),
кластеризовать плотностно (HDBSCAN), LLM только подписывает кластеры. Без
LLM-суждения о группировке.

**Меняемые компоненты.** Новая зависимость (эмбеддинги + `hdbscan`/`scikit-learn`); `cluster_profiles` → векторизация+кластеризация; промпт-лейблинг.
**Плюсы.** Детерминированно, без rate-limit на кластеризацию; масштабируется.
**Минусы.** Группирует по **теме**, не по поставке (изучение §2, категория B); тюнинг `min_cluster_size`/метрики; новая зависимость + эмбеддинг-инфра; ось не совпадает с целью заказчика.
**Усилия.** ~2 дня.
**Риски.** Высокие по релевантности — тематическая близость ≠ единица поставки. Не рекомендуется как основной; возможен как сигнал внутри A (пред-группировка).

## Концепт E — Taxonomy + карта компонентов (Правильное+, если есть данные)

**Суть.** A, но зоны привязываются к реальной имплементационной поверхности из
`capabilities.md`/карты кода: `derive_taxonomy` получает список движков/модулей и
группирует по «какой компонент затрагивается», а не только по семантике.

**Меняемые компоненты.** Как A + вход `capabilities.md` в `derive_taxonomy`.
**Плюсы.** Ось «одна итерация» самая точная (общий компонент = общая поставка); зоны маппятся на код.
**Минусы.** Требует полной карты компонентов — сейчас `capabilities.md` частичен (маппит пайплайны, не UI, см. память проекта). Без неё деградирует до A.
**Усилия.** A + ~1 день на карту.
**Риски.** Средние — зависит от полноты карты. Хороший **апгрейд A на будущее**, не для первой реализации.

---

## Рекомендация (Explicit Rejection)

**Принять Концепт A (taxonomy-first, 3 стадии)** — единственный, эмпирически
подтверждённый на реальном бэклоге и решающий обе проблемы (качество + replay).

- **B** — валидный MVP A; если нужен результат быстрее, реализовать A без `slice_zone` первой итерацией, slice — второй.
- **C** — отклонить как целевое (жёсткий энум, без boundary/нарезки); допустимо как экстренная затычка на часы.
- **D** — отклонить как основное (тематическая ось ≠ поставка); опц. сигнал внутри A.
- **E** — отложить до полной `capabilities.md`; апгрейд A, не замена.

Рекомендуемый путь: **A с фазировкой B→slice** (сначала derive+assign, затем slice),
`continue-as-new` в workflow для тримминга истории.
