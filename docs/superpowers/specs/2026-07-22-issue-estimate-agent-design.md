# Design: AI-агент оценки задач — команда `/estimate`

Дата: 2026-07-22
Статус: одобрен к реализации
Репозиторий: этот

## Цель и мерило успеха

Когда по Issue уже собран контекст и подготовлен план, человек должен получать
оценку трудоёмкости по единому набору правил, а не по устной договорённости
внутри команды.

Мерило успеха — сценарий демонстрации целиком:

1. Issue существует в репозитории.
2. Человек пишет комментарий `/estimate`.
3. Агент ставит реакцию 👀 на этот комментарий (подтверждение, что команда
   увидена, до начала долгого расчёта).
4. Запускается процесс оценки.
5. Агент публикует комментарий с оценкой и обоснованием: из чего сложилось
   число, какие коэффициенты и надбавки применены, на каком контексте
   основан расчёт.
6. Повторный `/estimate` даёт новую оценку с учётом контекста, появившегося
   с прошлого раза.

## Границы

**В объёме:**

- Оценка одиночного Issue: классификация типа работы → декомпозиция
  снизу-вверх → Function Points как cross-check → PERT (O/M/P) → риски и
  надбавки за недостающие данные → sanity bounds → нормировка по грейдам →
  Story Points и confidence.
- Обезличенная методология как документ репозитория плюс её исполняемая
  копия в конфиге.
- Команда `/estimate` в комментарии, реакция-подтверждение, публикация
  результата, переоценка по повторному вызову.
- Сбор контекста: тело Issue, лейблы, тред комментариев, артефакты ветки
  `research/issue-<n>` или `bug/issue-<n>`, если она существует.

**Вне объёма:**

- Оценка эпиков (сумма вложенных Issue, epic overhead, критический путь).
- Автодекомпозиция на подзадачи и создание sub-issues.
- Чтение исходного кода репозитория для оценки объёма правок (упирается в
  нерешённый вопрос доступа воркера к коду и Repowise, см.
  [ARCHITECTURE.md](../../ARCHITECTURE.md)).
- Сравнение с оценкой, проставленной человеком, и с фактической
  трудоёмкостью по факту закрытия.

## Обезличивание исходной методологии

Исходный документ приходит из внешнего корпоративного контура. В репозиторий
попадает только методология.

Удаляется: фамилия из заголовка, название компании и её продуктов, ссылки на
корпоративные Jira и Confluence, номера тикетов-примеров, названия внутренних
платформ и репозиториев, внутренние пути установки агента, раздел «См. также»
со ссылками на внутреннюю базу знаний, статистика по внутренней выборке задач.

Раздел «Бизнес-эффект» (таблица нормативов времени «без агента / с агентом»)
удаляется целиком: это внутренние показатели, а не методология.

Раздел про интеграции с конкретными корпоративными системами заменяется
нейтральной формулировкой «внешние источники контекста».

Раздел о смежных методологиях сохраняется без привязки к компании: RICE,
WSJF, ROI, Story Points по Фибоначчи — общеизвестные подходы.

## Архитектура

### Команда и запуск

`webhook/main.py` в ветке `issue_comment.created` дополнительно распознаёт
команду. Комментарий считается командой, если его **первая непустая строка**
после `strip()` равна `/estimate` без учёта регистра. Строка, начинающаяся
с `>` (цитирование), командой не считается. Комментарии от ботов отсекаются
существующим guard'ом.

Комментарий-команда **не** транслируется в сигнал `user_comment`: команда —
не ответ на уточняющий вопрос intake gate.

Запуск — отдельный workflow `IssueEstimation`, никак не связанный с
жизненным циклом `IssueLifecycle`. Причина: `IssueLifecycle` завершается
после приоритизации (а на спаме, дубликате и консультации — ещё раньше),
и через неделю сигналить в него было бы некуда.

Workflow ID = `estimate-{repo}-{issue_number}-{comment_id}`.

Следствия такого ID:

- Переоценка (шаг 6 сценария) — просто новый прогон от нового комментария.
  Никаких сигналов, очередей и вечноживущих workflow.
- Повторная доставка одного и того же вебхука GitHub даёт тот же ID, и
  Temporal отклоняет второй старт — идемпотентность бесплатно.
- Полная история всех оценок по Issue видна в Temporal UI.

Реакция 👀 ставится не в вебхуке, а первой activity workflow: вебхук остаётся
чистым транспортным слоем, как и для остальных событий. Задержка между
командой и реакцией равна времени старта workflow.

### Workflow `IssueEstimation`

| Шаг | Activity | Таймаут | Retry | Содержание |
|-----|----------|---------|-------|------------|
| 1 | `ack_estimate_command` | 30 с | 3 | реакция 👀 на комментарий-команду |
| 2 | `collect_estimation_context` | 120 с | 3 | Issue, лейблы, тред, артефакты ветки |
| 3 | `extract_estimation_facts` | 180 с | 3 | LLM + Instructor → факты |
| 4 | `compute_estimate` | 30 с | 1 | детерминированный расчёт |
| 5 | `post_estimate_comment` | 30 с | 3 | комментарий + лейбл `estimated` |

Шаг 4 не ретраится: он детерминирован и не ходит в сеть, повтор дал бы тот же
результат.

Исключение на любом шаге → `post_estimate_error`: комментарий с указанием
стадии, на которой сломалось, и реакция 😕 на комментарий-команду.

### Сбор контекста (шаг 2)

Источники по убыванию ценности:

1. Заголовок, тело, лейблы Issue.
2. Тред комментариев: без ботовых, без самих команд `/estimate`, без
   предыдущих комментариев-оценок.
3. Артефакты ветки. Проверяется существование `research/issue-<n>`, затем
   `bug/issue-<n>` (тем же `branch_exists`, что уже используется в
   `duplicate_check`). Если ветка найдена — читаются известные пути из
   модели данных артефактов: `docs/bft/issue-<n>-blueprint.md`,
   `docs/bft/issue-<n>-debate.md`, `docs/bft/issue-<n>-recommendations.md`,
   `docs/research/issue-<n>-sa-spec.md`, `docs/bugs/issue-<n>-diagnosis.md`.
   Отсутствующий файл — не ошибка.

Лимиты, чтобы контекст не разрастался неограниченно: до 50 комментариев треда
и до 20 000 символов на тред; до 20 000 символов на один артефакт и до
60 000 символов на все артефакты суммарно. Обрезка — с конца, с явной
пометкой в тексте.

Ветка, из которой взяты артефакты, попадает в вывод: человек должен видеть,
на чём основана оценка.

### Извлечение фактов (шаг 3)

Модель — `MODEL_CLASSIFY` (более сильная): оценка требует рассуждения, а не
классификации по шаблону. Схема ответа задаётся Pydantic через Instructor,
как в остальных стадиях.

LLM возвращает **только факты, не расчёт**:

- тип работы: `copy_existing` | `documentation` | `research` |
  `new_development` | `enhancement` | `deployment`;
- тип артефакта: `bugfix` | `validation` | `subtask` | `new_module`;
- часы на каркас;
- список единиц работы, каждая — название, часы, обоснование;
- часы на интеграцию;
- Function Points и часов на один FP;
- уровень достаточности данных: `insufficient` | `minimal` | `sufficient` |
  `complete`;
- четыре флага: есть ли критерии приёмки, перечислены ли зависимости,
  согласован ли контракт API, указан ли класс обрабатываемых данных;
- ключи применимых рисков: `personal_data`, `cross_team`,
  `undocumented_external_api`, `unagreed_api_contract`, `security_unreviewed`;
- открытые вопросы — что уточнить, чтобы сузить диапазон;
- краткое текстовое обоснование декомпозиции.

Промпт явно требует давать часы **как для новой разработки с нуля**.
Коэффициент типа работы накладывает код. Без этого требования скидка за
«копирование готового» применилась бы дважды: один раз в головах модели при
проставлении часов, второй раз кодом.

Поля «итоговая оценка» в схеме нет — модель структурно не может вернуть
готовое число в обход расчёта.

## Правила расчёта (шаг 4)

Чистая функция, без сети и без обращения к модели. Все числовые параметры —
из `config/estimation-rules.toml`.

```
base    = каркас + Σ(часы единиц работы) + интеграция
tests   = base × test_share.ratio
M_base  = (base + tests) × work_type[тип работы]
FP_base = fp_count × fp_hours_per_point × work_type[тип работы]
```

Cross-check двух методов:

```
divergence = |M_base − FP_base| / min(M_base, FP_base)
```

Если `divergence > cross_check.max_divergence` — расчёт останавливается.
Итоговые числа **не публикуются**. Публикуется расхождение, оба базовых
расчёта и открытые вопросы. Это прямое требование методологии: при
расхождении методов больше чем вдвое ошибка ищется, а не замазывается
усреднением.

PERT:

```
O = M_base × pert.optimistic_factor
M = M_base × (1 + Σ risk[r].min)
P = M_base × pert.pessimistic_factor × (1 + Σ risk[r].max)
                                     × (1 + Σ применённых надбавок)

E      = (O + 4×M + P) / 6
SD     = (P − O) / 6
буфер  = E + pert.buffer_k × SD
```

Надбавки за недостающие данные применяются только к пессимистичному
сценарию и только те, чей флаг ложен: нет критериев приёмки, не перечислены
зависимости, не согласован контракт API, не указан класс данных.

Story Points: `E / day.hours` переводится в дни, `E / story_points.hours_per_point`
округляется до ближайшего значения шкалы Фибоначчи. Если оценка выходит за
верх шкалы — выводится верхнее значение со знаком `+` и рекомендация
декомпозировать.

Грейды: `M` умножается на `grade.middle`, `grade.senior`, `grade.senior_ai`.

Sanity bounds: `E` в днях сравнивается с коридором для типа артефакта.
Выход за коридор **не меняет число** — поднимается флаг с рекомендацией из
методологии (например, «дольше трёх дней — это уже не багфикс, а доработка»).

Confidence выводится из уровня достаточности данных: `complete` и `sufficient`
дают `high`, `minimal` — `medium`, `insufficient` — `low`. Если при этом
`divergence` превышает половину порога, confidence понижается на ступень.

Внутренние проверки перед выводом: `O < M < P`, все часы строго положительны,
сумма единиц работы не равна нулю. Нарушение любой — ошибка расчёта, а не
молчаливая корректировка. Произвольный множитель «для подгонки» структурно
невозможен: такого поля нет ни в схеме LLM, ни в конфиге.

## Конфиг `config/estimation-rules.toml`

Секции и ключи (значения — из методологии, средние по диапазонам там, где
методология даёт диапазон):

- `work_type` — коэффициент по типу работы, шесть ключей.
- `test_share.ratio` — доля тестов от объёма единиц работы.
- `missing_data_penalty` — четыре надбавки к пессимистичному сценарию.
- `risk` — пять рисков, каждый парой `min`/`max`.
- `sanity_bounds` — четыре типа артефакта, каждый парой границ в днях, и
  текст рекомендации при выходе за коридор.
- `grade` — три коэффициента нормировки труда.
- `pert` — `optimistic_factor`, `pessimistic_factor`, `buffer_k`.
- `cross_check.max_divergence`.
- `day.hours`, `story_points.hours_per_point`, `story_points.scale`.

Числа живут в TOML — он источник правды для расчёта. Те же числа приведены в
таблицах методологии для читаемости. Чтобы копии не разъехались, каждая
числовая таблица методологии содержит колонку с точным путём ключа в TOML
(например `work_type.copy_existing`), а тест сверяет таблицы с конфигом.

## Формат вывода (шаг 5)

Комментарий содержит, сверху вниз:

1. Итог: реалистичная оценка в днях, Story Points, confidence.
2. Таблица двух методов — декомпозиция и FP — с базовыми числами и
   расхождением между ними.
3. Разбор декомпозиции: каркас, единицы работы с часами и обоснованием,
   тесты, интеграция, применённый коэффициент типа работы.
4. Таблица PERT: O / M / P в днях, `E`, буфер.
5. Таблица по грейдам.
6. Расшифровка **каждого** применённого риска и **каждой** надбавки: что
   именно применено, с каким коэффициентом и почему.
7. Sanity-флаги, если есть.
8. Что уточнить, чтобы сузить диапазон.
9. Источники контекста: «описание + N комментариев + ветка `research/issue-N`
   (M файлов)».

Пункты 2, 3, 6 и 9 — это и есть «артефакты обоснования» из сценария
демонстрации: по ним видно, откуда взялось число.

Лейбл `estimated` идемпотентен, при переоценке ставится повторно без вреда.

- `webhook/main.py` — распознавание команды, старт `IssueEstimation`,
  подавление сигнала `user_comment` для команд.

- `worker/activities.py` — пять новых activity плюс `post_estimate_error`.
- `worker/worker.py` — регистрация нового workflow и activity.
- `shared/workflow_types.py` — dataclass'ы `EstimateRequest`,
  `EstimationContext`, `EstimateResult`.
- `prompts/system_estimate_extract.md` — новый промпт.
- `config/estimation-rules.toml` — новый конфиг.
- `docs/methodology/task-estimation.md` — обезличенная методология.

Существующий поток `IssueLifecycle` не меняется.

## Тестирование

- Расчётная функция: коэффициенты по типам работы, каждая надбавка отдельно,
  каждый риск отдельно, арифметика PERT, остановка по cross-check, флаги
  sanity bounds, округление Story Points, инвариант `O < M < P`, понижение
  confidence при повышенном расхождении.
- Парсер команды: регистр, ведущие пробелы, `/estimate` в середине текста не
  срабатывает, цитата `> /estimate` не срабатывает, команда с хвостом.
- Workflow целиком через `WorkflowEnvironment` с мок-activity: happy path
  и путь ошибки.
- Новые вызовы GitHub уважают `DRY_RUN`.
- Синхронизация: числа в TOML совпадают с таблицами методологии.

## Риски реализации

Основной риск — качество извлечения единиц работы моделью: при скудном
описании она склонна выдумывать правдоподобную декомпозицию. Защита
трёхслойная: cross-check с FP ловит грубые расхождения, sanity bounds ловят
выход за реалистичный коридор, confidence и блок «что уточнить» честно
сообщают человеку, что вход был бедным. Ни один из трёх слоёв не подгоняет
число — все три только сигнализируют.
