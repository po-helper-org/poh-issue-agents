# Журнал архитектурных решений (ADR-lite)

Зафиксированные решения из проработки, чтобы завтра не переоткрывать
обсуждённое. Формат: Решение — Причина — Отклонённые альтернативы.

## ADR-1. Оркестратор — Temporal, не GitHub Actions
**Причина:** нужен постоянный сервис ("поднял и работает"), durable
execution длинных research-прогонов, неограниченное ожидание сигнала
человека (Issue висит в бэклоге неделями), устранение гонок между
параллельными Actions.
**Отклонено:** чистый GitHub Actions (исходная реализация) — эфемерные
раннеры, гонки между workflow на один лейбл, ручной парсинг состояния в
HTML-маркерах.

## ADR-2. Деплой — docker-compose, не Kubernetes
**Причина:** цель "как pr-agent-glm-setup" — один `docker-compose up`.
Совместимость с Dokploy.
**Отклонено:** KAgent — Kubernetes-контроллер, требует кластера, не
поднимается в Dokploy напрямую; governance для роя разнородных агентов не
нужен для одного типа агента; нет durable-ожидания произвольного
бизнес-сигнала.

## ADR-3. LLM — GLM через z.ai, не Claude по подписке
**Причина:** GLM через z.ai — pay-as-you-go, без лицензионной
неоднозначности OAuth-токена подписки Claude в CI (Anthropic запрещает
подписочный токен вне "ordinary individual usage", энфорсит блокировками).
z.ai держит и OpenAI-, и Anthropic-совместимый эндпоинты.
**Отклонено:** Claude Pro/Max подписка в CI — юридически серая зона,
риск блокировки; API-ключ Anthropic — платно за токены (GLM дешевле).

## ADR-4. Priority Scoring — LLM извлекает, код считает
**Причина:** детерминизм. "Спроси LLM оценить приоритет 1-10" даёт разные
числа на одном Issue. LLM извлекает атрибуты по рубрике → Python считает
по фиксированной формуле → воспроизводимо и объяснимо.
**Отклонено:** прямая оценка приоритета моделью — недетерминированно.

## ADR-5. Структурированное извлечение — Instructor + Pydantic
**Причина:** типобезопасность, авто-retry при невалидной схеме, убирает
ручной json.loads()+try/except. Индустриальный стандарт под паттерн
"structured extraction".
**Отклонено:** ручной парсинг маркеров `[[MARKER]]` (исходная версия) —
хрупко, требует fail-open обработки.

## ADR-6. Баг и фича — раздельные пайплайны
**Причина:** у бага нет бизнес-критериев приёмки и предмета для
стратегических дебатов. po-helper и deb8flow для бага избыточны.
**Отклонено:** единый пайплайн для всего — тратит дорогие стадии
(БФТ, дебаты) там, где они не нужны.

## ADR-7. po-helper — только критерии приёмки
**Причина:** чёткое разделение ролей. po-helper описывает "что значит
готово" (Definition of Done, границы, UJM, интерфейсы, НФТ), SA-helper —
"как делать". Смешение размывает ответственность.
**Отклонено:** po-helper пишет и требования, и реализацию — пересечение
с SA-helper.

## ADR-8. Blueprint между БФТ и SA-helper
**Причина:** поймать несостыковки данных/контрактов между компонентами
на верхнем уровне ДО детального анализа. Service Blueprint + IDEF0 —
уровень контракта ("что входит/регулирует/выходит/кто исполняет"), не
реализации. Раздел "Несогласованности" — явный вход для дебатов и SA-helper.
**Отклонено:** сразу в детальный системный анализ — несостыковки всплывают
поздно и дорого.

## ADR-9. Дорогие стадии — только ручной запуск человеком
**Причина:** экономия ресурсов. Автоматом идут только бесплатные/дешёвые
фильтры. research/bug стартуют по лейблу человека после просмотра
приоритета.
**Отклонено:** полная автоматизация до кода — жжёт ресурсы на нецелевые
запросы.

## ADR-10. Duplicate check — один вызов на всех кандидатов
**Причина:** дёшево. Один LLM-вызов оценивает всех кандидатов сразу, не по
вызову на кандидата. Решение по порогу — детерминированный код.
**Отклонено:** вызов на каждого кандидата — дорого и не нужно.

## ADR-11. Intake gate на дешёвой модели, отдельно от классификации
**Причина:** спам/расплывчатость отсекаются раньше и дешевле полной
классификации. Gate на GLM-air, классификация на GLM-5.2.
**Отклонено:** одна модель на всё — платим за сильную модель на явном
мусоре.

## ADR-12. OpenHands + pr-agent для стадии кода, вне ядра
**Причина:** OpenHands — self-hosted, sub-agent delegation, event stream ≈
issue-тред. pr-agent — независимый второй ревьюер поверх PR. OpenHands
держим отдельным сервисом (docker.sock = root на хосте, изоляция).
**Отклонено:** Copilot coding agent (облако MS, биллинг USD, для MTS
неприменим); SWE-agent (Python-центричный, библиотека а не платформа);
gh-aw для кода (не поддерживает OAuth-подписку, но кандидат на будущее).
