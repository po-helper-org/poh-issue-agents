# Требования: Issue Agent Service

Статус: зафиксировано для старта разработки
Версия: 1.0

Платформа автономной обработки GitHub Issues от момента открытия до передачи
в разработку. Цель — реагировать на **любой** входящий Issue и, не тратя
ресурсы на нецелевые запросы, довести его до состояния, в котором инженер
получает задачу с готовой аналитикой (критерии приёмки, компонентная карта,
техническая спецификация) вместо сырого текста.

---

## 1. Функциональные требования

### FR-1. Zero-cost предфильтры (без вызова LLM)
- FR-1.1. Issue от аккаунтов типа `Bot` (GitHub App) отсекаются до любого
  вызова модели. Лейбл `bot-authored`, обработка останавливается.
- FR-1.2. Денайлист известных сервис-аккаунтов (dependabot, renovate,
  snyk-bot и т.п.) + суффикс `[bot]` + маркеры автогенерации в теле.
- FR-1.3. Keyword/CVE-фильтр на security-чувствительные Issue. При
  срабатывании — лейбл `security-sensitive`, комментарий с рекомендацией
  перенести в GitHub Private Vulnerability Reporting, дальнейшая
  автоматическая обработка останавливается.
- FR-1.4. Основная защита security — на уровне репозитория (Private
  Vulnerability Reporting), keyword-фильтр — только подстраховка.

### FR-2. Intake Gate (дешёвая модель)
- FR-2.1. Классификация входящего Issue: `SPAM` / `VAGUE` / `SUFFICIENT`.
- FR-2.2. `SPAM` → комментарий с причиной, лейбл `spam`, автозакрытие.
- FR-2.3. `VAGUE` → 1-3 уточняющих вопроса, лейбл `needs-clarification`,
  ожидание ответа пользователя.
- FR-2.4. Цикл уточнения: ответ пользователя пере-запускает gate с учётом
  всей переписки. Потолок — 2 раунда, после чего эскалация на человека
  (`needs-human-triage`).
- FR-2.5. `SUFFICIENT` → переход к полной классификации.
- FR-2.6. При сомнении gate выбирает менее блокирующий исход (SPAM→VAGUE,
  VAGUE→SUFFICIENT) — консервативность в сторону пропуска, не отсечения.

### FR-3. Классификация (сильная модель)
- FR-3.1. 4-way классификация: `EXISTING` / `CONSULTATION` / `BUG` /
  `FEATURE`.
- FR-3.2. `EXISTING` → ответ "как использовать", закрытие содержательным
  ответом.
- FR-3.3. `CONSULTATION` → варианты решения (Ситуация/Ограничение/Решение),
  закрытие ответом.
- FR-3.4. `BUG` → лейбл `advisor:bug`, переход дальше.
- FR-3.5. `FEATURE` → лейбл `advisor:feature-request`, переход дальше.
- FR-3.6. Классификация опирается на `capabilities.md` (известный
  функционал) — источник правды для отличия существующего от нового.

### FR-4. Duplicate Check (дешёвая модель, один вызов на всех кандидатов)
- FR-4.1. Поиск кандидатов среди открытых и закрытых Issue и PR через
  `gh` CLI.
- FR-4.2. Один LLM-вызов оценивает вероятность дубликата для всех
  кандидатов сразу (не по вызову на кандидата).
- FR-4.3. Решение по фиксированному порогу (детерминированное):
  - `>= 0.85` → автозакрытие, ссылка на оригинал + на ветку с уже
    наработанным контекстом (research/bug), если она есть.
  - `0.5–0.85` → лейбл `possible-duplicate`, без закрытия, видимо на
    следующих стадиях.
  - `< 0.5` → не дубликат; кандидаты кэшируются для переиспользования
    research-стадией.

### FR-5. Priority Scoring
- FR-5.1. **Строгое разделение**: LLM извлекает структурированные атрибуты
  по рубрике (impact, time_criticality, risk_reduction, effort,
  okr_alignment) — детерминированный код считает приоритет.
- FR-5.2. Формула фиксирована в конфиге:
  `score = (impact + time_criticality + risk_reduction) × okr_multiplier / effort`.
- FR-5.3. Маппинг score → `P0`/`P1`/`P2`/`P3` по порогам из конфига.
- FR-5.4. Override: критичный баг (потеря данных/безопасность/полная
  недоступность) принудительно `P0` независимо от формулы.
- FR-5.5. Результат — лейбл `priority:P0..P3` + комментарий с полной
  разбивкой расчёта и последствиями (кто исполняет, риски, влияние на цели).
- FR-5.6. При одинаковых входных атрибутах результат детерминирован.

### FR-6. Research Pipeline (только FEATURE, запуск человеком)
- FR-6.1. Запускается по лейблу `research-me`, установленному человеком
  после просмотра приоритета.
- FR-6.2. Guard: не запускается без лейбла `advisor:feature-request`.
- FR-6.3. Стадия po-helper: БФТ-документ (.docx), роль сужена **только** до
  критериев приёмки (Definition of Done, входит/не входит в объём, UJM,
  требования к интерфейсам, нефункциональные требования). Без описания
  реализации.
- FR-6.4. Стадия контекста (Repowise + gh): дозаполнение БФТ секциями
  "Соответствие стратегии", "Связанные issue", "История/ранее решённое",
  "Обзор рыночных решений".
- FR-6.5. Стадия Blueprint (Repowise): Service Blueprint (действия
  пользователя → frontstage → backstage → процессы поддержки) + диаграмма
  Mermaid + формализация узлов по IDEF0 (Input/Control/Output/Mechanism).
  Обязательный результат — раздел "Несогласованности и открытые вопросы".
- FR-6.6. Стадия дебатов (deb8flow): мультиагентные дебаты по обогащённому
  БФТ (.docx на входе), на выходе протокол диалога + рекомендации (.md).
- FR-6.7. Стадия SA-helper: детальная техническая спецификация. Обязана
  явно закрыть или перенести как риск каждый пункт из "Несогласованностей"
  Blueprint.
- FR-6.8. Все артефакты — в git-ветку `research/issue-<n>`, комментарий со
  ссылками в Issue, лейбл `research:done`.

### FR-7. Bug Pipeline (только BUG, запуск человеком)
- FR-7.1. Запускается по лейблу `bug-me`. Guard: требует `advisor:bug`.
- FR-7.2. Поиск дублей/регрессии + SA-helper формирует диагноз
  (гипотеза первопричины, шаги воспроизведения, Definition of Done для
  фикса, оценка риска).
- FR-7.3. **Без po-helper и без дебатов** — осознанное решение: баг не
  требует бизнес-критериев приёмки и стратегических дебатов.
- FR-7.4. Артефакт — в ветку `bug/issue-<n>`, лейбл `bug:diagnosed`.

### FR-8. Передача в разработку (следующая система, вне ядра)
- FR-8.1. По лейблу `build-me` (человек) — вызов OpenHands resolver для
  автономного написания кода и открытия draft PR.
- FR-8.2. pr-agent как независимый второй ревьюер поверх PR от OpenHands.
- FR-8.3. OpenHands остаётся **отдельным** сервисом со своим sandboxing
  (docker.sock), не частью ядра платформы.

### FR-9. Диалог и обратная связь
- FR-9.1. Точки решения человека реализованы как сигналы/лейблы:
  `research-me`, `bug-me`, `build-me`.
- FR-9.2. Ручная смена классификационного лейбла человеком должна
  пересчитывать зависимые стадии (override классификации/приоритета).
- FR-9.3. (Отложено) Сбор сигналов рассогласования (человек изменил
  классификацию/приоритет) для калибровки промптов и весов.

## 2. Нефункциональные требования

- NFR-1. **Self-hosting.** Вся платформа разворачивается на собственной
  инфраструктуре (docker-compose). Без обязательной зависимости от
  облачных сервисов зарубежных вендоров в проде.
- NFR-2. **Экономия ресурсов.** Дорогие стадии (research/bug) запускаются
  только вручную человеком после прохождения бесплатных/дешёвых фильтров.
  На 100% Issue выполняются только предфильтры + gate.
- NFR-3. **Детерминизм решений.** Приоритизация и пороги дубликатов —
  детерминированный код, не "суждение LLM на глаз".
- NFR-4. **Durable execution.** Долгие стадии (research, до 60 мин)
  переживают рестарт воркера. Ожидание сигнала человека — неограниченное
  по времени (Issue может неделями ждать в бэклоге).
- NFR-5. **Идемпотентность.** Повторный webhook по тому же Issue не создаёт
  дублирующую обработку (workflow ID = `issue-<repo>-<n>`).
- NFR-6. **Наблюдаемость.** Состояние каждого Issue и стадии видно в
  Temporal UI без чтения сырых логов.
- NFR-7. **Provider-agnostic по модели.** Смена LLM (GLM ↔ другой
  провайдер ↔ self-hosted) — через переменные окружения, без изменения
  оркестрации.
- NFR-8. **Graceful failure.** Сбой вызова LLM/API не оставляет Issue в
  молчании — либо retry, либо понятный комментарий об ошибке.

## 3. Технологические решения (зафиксировано)

| Слой | Решение | Обоснование |
|---|---|---|
| Оркестрация | Temporal (self-hosted, docker-compose) | Durable execution, ожидание сигнала неограниченно, идемпотентность по workflow ID |
| Транспорт событий | FastAPI webhook-приёмник + GitHub App | Один вход, трансляция webhook → старт/сигнал Temporal |
| LLM (дешёвые стадии) | GLM-4.5-air через z.ai (OpenAI-совм.) + Instructor/Pydantic | Структурированное извлечение с валидацией, детерминизм контракта |
| LLM (классиф./research) | GLM-5.2 через z.ai | Флагман, 1M контекст под SA-helper |
| Claude Code skills | `claude -p` + Anthropic-совм. эндпоинт z.ai | po-helper/SA-helper без изменений, меняется только backend |
| Детерм. расчёт | Чистый Python + TOML-конфиг | Воспроизводимость приоритета |
| Деплой | docker-compose (Dokploy-совместимо) | "Поднял и работает", без Kubernetes |
| Разработка кода (стадия 8) | OpenHands (отд. сервис) + pr-agent | Уже выбрано ранее, вне ядра |

### Отклонённые варианты (с причиной)
- **GitHub Agentic Workflows (gh-aw)** — не поддерживает подписочный
  OAuth-токен; философия "judgment, не детерминизм" конфликтует с Priority
  Scoring. Кандидат на стадию 8, не на ядро.
- **KAgent** — Kubernetes-контроллер, требует кластера (несовместим с
  Dokploy/docker-compose напрямую); governance для роя разнородных агентов —
  не нужен для одного типа агента на чётких стадиях; нет durable-ожидания
  произвольного бизнес-сигнала. Уместен был бы при наличии prod-K8s и задачи
  SRE-агентов.
- **Чистый GitHub Actions** — исходная реализация; отойти решено ради
  постоянного сервиса, durable execution и устранения гонок между
  параллельными workflow. Логика перенесена без изменений по смыслу.

---

## 4. Границы (что НЕ входит)

- Автономное написание кода (стадия 8, OpenHands) — отдельная система.
- Авто-синхронизация `capabilities.md` с кодовой базой — заполняется
  вручную на старте.
- Подключение реального источника OKR к priority-scoring — на старте
  `okr_alignment` может быть `unrelated`/из статичного файла.
- Метрики калибровки (FR-9.3) — после пилота.
