# Как кластеризовать бэклог под поставку — изучение

> **Дата:** 2026-07-15
> **Контекст:** первая версия consolidation (single-shot и map-reduce reduce над
> `proposed_mechanism`) дала 54 одиночки на 75 Issue. Это изучение — почему, какие
> есть практики группировки, и что делать.

---

## 0. Переформулировка цели (главное)

Заказчик уточнил: **кластер = единица поставки.** Цель — за одну техническую
итерацию и релиз закрыть максимум схожих требований. Значит ось группировки —
**«что реализуется и релизится вместе»**, а НЕ «похожая тема» и НЕ «одинаковая
формулировка механизма».

Это меняет всё. Прошлая версия кластеризовала по verbatim-`proposed_mechanism` —
неверная ось. Два Issue про JIRA («индексатор графа» и «массовые операции») имеют
РАЗНЫЕ строки механизма, но поставляются **одним движком JIRA-интеграции за одну
итерацию** — их надо в один кластер.

---

## 1. Почему текущий подход провалился (на реальных данных)

**Гранулярность.** `proposed_mechanism` извлекается пер-Issue и получается
уникальным почти у каждого («jira-indexator», «jira-sync gantt», «mass JIRA ops»,
«JIRA discovery»...). Модель видит 57 разных строк → делает 57 кластеров.

**Неверная ось.** «Одинаковый механизм» — не то же, что «поставляется вместе».
Нужна ось «общая имплементационная поверхность / зона возможностей».

**Открытая кластеризация — слабое место LLM.** Просьба «сгруппируй 57 элементов»
— это задача без якоря; glm сваливается в 1-кластер-на-элемент. Map-reduce не помог
и стал хуже: батчи разнесли связанные Issue, merge их не собрал (потеряли 3 пары,
что нашёл single-shot).

**Доказательство обратного — бэклог ХОРОШО кластеризуется** (глазами), значит
проблема в методе, не в данных. Естественные кластеры видны сразу (§5).

---

## 2. Практики группировки задач (обзор, с оценкой под нашу цель)

| Практика | Суть | Плюс | Минус под «поставку» |
|----------|------|------|----------------------|
| **A. Taxonomy-first / affinity (card sorting, DDD bounded contexts, agile epic/theme)** | Сверху задать малый набор зон-возможностей, отнести каждую задачу в зону | Коарс, стабильно, человекочитаемо, ложится на релиз-инкременты | Нужно вывести хорошую таксономию один раз |
| **B. Эмбеддинги + плотностная кластеризация (HDBSCAN, k-means)** | Векторизовать текст Issue, кластеризовать по близости | Автоматично, без LLM-суждения | Группирует по ТЕМЕ, не по поставке; нестабильно; тюнинг eps/k |
| **C. Зависимостно-компонентная (shared modules, build order)** | Группировать по тому, какие модули/файлы затрагивает решение | Ближе всего к «одна итерация» | Нужна карта компонентов; у нас её пока нет |
| **D. Граф-сообщества (Louvain, label propagation)** | Рёбра Issue↔Issue (общий домен / компонент / ручные «Связано»), детект сообществ | Использует уже нарисованные связи | Нужны рёбра; качество зависит от их полноты |
| **E. LLM попарно + транзитивное замыкание** | «Эти два поставятся вместе?» по парам → граф → кластеры | Точно, контролируемо | O(n²) вызовов — дорого на 75 Issue |
| **F. MECE-декомпозиция / дерево возможностей** | Дерево capability, задачи в листья | Строго, без пересечений | Ручной каркас |

**Ключевой вывод обзора:** для нашей цели верхний претендент — **A
(taxonomy-first)**, усиленный сигналами из **C (компоненты)** и опционально **D
(граф связей)**. B/E/F — вспомогательные.

Почему A, а не открытая кластеризация: LLM **надёжно классифицирует** в заданный
малый словарь, но **плохо изобретает** группировку с нуля. Мы сделали второе —
надо первое.

---

## 3. Рекомендация — taxonomy-first по оси поставки

Три стадии вместо одного reduce:

### Стадия 1 — вывести таксономию зон-поставки (1 вызов на весь бэклог)
Один LLM-проход по ВСЕМ заголовкам + profile'ам (компактно) → **8–12 зон
возможностей**, каждая:
- имя (`jira-engine`, `memory-core`, `bft-pipeline`, `ingestion-inbox`, …);
- «что закрывает одна итерация этой зоны» (граница поставки);
- ключевая имплементационная поверхность (какой модуль/движок).

Это НЕ пер-Issue механизмы — это взгляд на весь набор сразу, малый фиксированный
словарь. Именно здесь чинится гранулярность.

### Стадия 2 — назначить Issue в зоны (пер-Issue, fan-out)
На каждый Issue — классификация в **primary-зону** (+ secondary для сквозных).
Это надёжная LLM-операция (выбор из 8–12), а не открытая кластеризация.
Параллельно, дёшево, детерминированнее. Заодно чинит replay-масштаб: результат —
компактный ярлык, не тяжёлый объект.

### Стадия 3 — нарезать зону на итерации (по размеру + зависимостям)
Зона в 14 Issue (JIRA) — это НЕ один релиз. Внутри зоны резать на
итерации-инкременты (MVP / MVP+1) по:
- порядку зависимостей (что должно приехать первым);
- потолку размера итерации (≈ N Issue / бюджет усилий).

Это прямо ложится на #97 (итерационное дробление БФТ MVP/MVP+1) и #133
(декомпозиция эпика на зоны).

### Инварианты (сохранить)
- **#111 target-guard** внутри зоны: одинаковая функция, но разная цель (запуск vs
  перевод на архитектуру) — разные инкременты, не слишком-сливать.
- **Множественное членство + роли**: сквозной Issue (напр. #75 freshness) —
  primary в memory-core, secondary в JIRA.
- **Провенанс/anchors** на каждом требовании — как есть.

---

## 4. Почему это чинит оба провала

- **Качество группировки:** ось = зона поставки (коарс), словарь фиксирован →
  модель не плодит одиночки. Классификация вместо открытой кластеризации.
- **Replay-масштаб:** стадия 2 пер-Issue, результат — ярлык (не 61 тяжёлый объект
  через историю). Плюс `continue-as-new` при желании тримит историю.

## 6. Дельта реализации (небольшая, ложится на текущий код)

Заменить один `cluster_profiles` reduce на:
1. `derive_taxonomy(profiles) -> list[DeliveryZone]` — 1 activity, весь бэклог
   компактно, сильная модель.
2. `assign_zone(issue_profile, taxonomy) -> ZoneAssignment` — activity, fan-out
   пер-Issue (как profile-extract).
3. `slice_zone(zone, members, profiles) -> list[Increment]` — опционально, режет
   крупную зону по зависимостям/размеру.

Переиспользует fan-out-инфру и `llm.extract`. `synthesize_unifying_issue` и PR-часть
почти без изменений (работают над инкрементами вместо кластеров). Плюс
`continue-as-new` в workflow для тримминга истории (replay-фикс).

**Сигналы для оси поставки** (усилить зоны):
- общий компонент/модуль (из `capabilities.md` / карты кода) — стадия 1 должна
  привязывать зону к имплементационной поверхности;
- зависимости между зонами (memory-core раньше того, что на нём стоит);
- потолок размера итерации.

## 7. Что дальше

Это дизайн-итерация, не правка. Порядок:
1. Спека FNR-3 «delivery-taxonomy clustering» (эта записка — вход).
2. Реализация 3 стадий по TDD (как FNR-2).
3. Прогон — ожидаемо ~8 зон вместо 54 одиночек.

Быстрый дешёвый эксперимент до полной реализации: прогнать ТОЛЬКО стадию 1
(derive_taxonomy) над готовыми 61 профилями (они уже извлечены) + стадию 2
(assign) — проверить, что зоны выходят как в §5, за ~15 мин и без нового пайплайна.

## Приложение: результаты дешёвого эксперимента (2026-07-15)

Прогнан taxonomy-first БЕЗ пайплайна (standalone, `llm.extract`, ~6 вызовов).

**Стадия 1+2 (derive+assign):** 8 зон, 70 Issue назначены (0 unassigned), 5
обсуждений исключены. Размеры зон: jira-pm-integration 13, planning-execution 14,
bft-research-cycle 11, memory-graph-core 7, devops-platform 7, external-signals 6,
ui-visualization 6, ai-analytics-intelligence 6. (vs 54 одиночки старого подхода.)

**Стадия 3 (slice):** planning-execution → 4 инкремента (MVP..MVP+3, ~3-5 Issue,
по зависимостям фундамент→надстройка); jira-pm-integration → 3 инкремента.

**Synth:** memory-graph-core и jira-pm-integration дали объединяющие Issue с
синтезом проблемы + механизмом + агрегатом требований (по пункту на Issue, `— from #N`).

Вывод: подход подтверждён на всех стадиях. Переход к FNR-3.
