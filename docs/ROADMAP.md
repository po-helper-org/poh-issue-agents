# Roadmap реализации: Issue Agent Service

Фазированный план от заготовки (текущее состояние) до полного цикла.
Каждая фаза заканчивается проверяемым результатом. Фазы упорядочены так,
чтобы как можно раньше получить работающий сквозной путь на дешёвых
стадиях, и только потом наращивать дорогие.

---

## Что уже готово в заготовке

- `docker-compose.yml` — 5 сервисов (Postgres, Temporal, Temporal UI,
  webhook, worker).
- `webhook/main.py` — приём webhook, трансляция в старт/сигнал Temporal.
- `worker/workflows.py` — `IssueLifecycle` со всеми точками ожидания.
- `worker/activities.py` — предфильтры, gate, классификация, дубликаты,
  приоритизация (перенесено из отлаженной логики на Actions). Стадии
  research/bug/openhands — заглушки `NotImplementedError`.
- `worker/llm.py` — Instructor + z.ai/GLM.
- `worker/github_client.py` — GitHub App аутентификация.
- `prompts/` — 4 системных промпта (advisor, intake_gate, priority_extract,
  blueprint).
- `config/priority-weights.toml` — веса и пороги.

---

## Фаза 0. Инфраструктура и первый webhook (день 1)

Цель: сервис поднимается, GitHub достукивается, workflow стартует.

- [ ] `docker-compose up --build` поднимает все 5 сервисов без ошибок.
- [ ] Зарегистрировать GitHub App (permissions: Issues r/w, Contents r/w;
      events: Issues, Issue comments). Сохранить App ID, Installation ID,
      private key.
- [ ] Заполнить `.env` из `.env.example`.
- [ ] Смонтировать private key как volume/secret (в `docker-compose.yml`
      пока НЕ прописано — добавить).
- [ ] Публичная доступность webhook: `cloudflared tunnel`/`ngrok` для
      разработки, reverse proxy на домене для прода.
- [ ] Проверка: открыть тестовый Issue → в Temporal UI (localhost:8080)
      появился workflow-инстанс.

**Чекпоинт Ф0:** новый Issue создаёт workflow, видимый в Temporal UI.

**Риск:** без публичного webhook GitHub физически не достучится — это
первое, что надо проверить, всё остальное можно тестировать только после.

## Фаза 1. Сквозной дешёвый путь (дни 1-3)

Цель: любой Issue проходит предфильтры → gate → классификацию →
дубликаты → приоритет и получает лейбл + комментарий. Это уже полезный
самостоятельный продукт.

- [ ] Проверить GLM через z.ai: Instructor возвращает валидные
      Pydantic-объекты на реальных вызовах (`intake_gate`, `classify`).
      **Первый практический тест GLM tool/schema-совместимости.**
- [ ] Прогнать предфильтры на тестовых Issue (бот, security-keyword).
- [ ] Прогнать gate на спаме / расплывчатом / нормальном Issue.
- [ ] Проверить цикл уточнения: VAGUE → вопрос → ответ пользователя
      (signal) → повторный gate → потолок 2 раунда → эскалация.
- [ ] Прогнать классификацию на всех 4 типах.
- [ ] Прогнать duplicate-check: создать заведомый дубликат, проверить
      пороги 85%/50%.
- [ ] Прогнать priority-scoring: проверить детерминизм (один Issue дважды →
      тот же приоритет) и override критичного бага.
- [ ] Заполнить `capabilities.md` реальным функционалом (иначе
      классификация не отличит существующее).

**Чекпоинт Ф1:** Issue доходит до `priority:PN` + комментарий с разбивкой,
полностью автономно, без единой ручной операции.

**Открытые вопросы Ф1:**
- Промпты написаны под старый парсинг маркеров `[[...]]`. С Instructor это
  избыточно — можно упростить, убрав инструкции "начни с маркера" (не
  блокирует, но чище).
- Калибровка порогов `priority-weights.toml`: прогнать на 10-15 реальных
  прошлых Issue, сверить с интуитивной расстановкой, подправить веса.
- Источник OKR для `okr_alignment`: на старте — статичный файл
  `docs/okr/current-quarter.md` или всегда `unrelated`. Реальное
  подключение (Cortex/Nexus/MCP) — позже.

## Фаза 2. Отказоустойчивость и границы (дни 3-4)

Цель: сбой не оставляет Issue в молчании.

- [ ] try/except вокруг LLM-вызовов в activities: при сбое — лейбл
      `advisor:error` + комментарий "автообработка не удалась, ожидай
      ручного разбора", а не тихое падение workflow.
- [ ] Проверить behavior при недоступности z.ai (таймаут/500): RetryPolicy
      отрабатывает, после исчерпания — понятный комментарий.
- [ ] Guard `possible-duplicate`: research/bug-стадии не стартуют, если
      висит `possible-duplicate` без `confirmed-not-duplicate`.
- [ ] Обработка отредактированных Issue (`issues.edited`): решить —
      переклассифицировать или игнорировать (сейчас только opened).
- [ ] Override классификации: ручная смена лейбла человеком пересчитывает
      зависимые стадии.

**Чекпоинт Ф2:** ни один сбой инфраструктуры/модели не оставляет Issue
без обратной связи в комментариях.

## Фаза 3. Bug Pipeline (дни 4-6)

Цель: BUG доходит до диагноза. Начинаем с багов — они проще (нет
po-helper, нет дебатов).

- [ ] Реализовать `run_bug_pipeline` (сейчас NotImplementedError):
      поиск дублей/регрессии + SA-helper диагноз.
- [ ] Решить механизм загрузки скилла SA-helper в worker-контейнер
      (`.claude/skills/` в образе или plugin install). **Блокер для
      всех claude -p стадий.**
- [ ] Проверить `claude -p` через Anthropic-совместимый эндпоинт z.ai:
      GLM корректно вызывает Write/Read инструменты.
- [ ] Артефакт `docs/bugs/issue-<n>-diagnosis.md` коммитится в ветку
      `bug/issue-<n>`, лейбл `bug:diagnosed`.

**Чекпоинт Ф3:** BUG с лейблом `bug-me` → диагноз + DoD в git-ветке +
комментарий со ссылкой.

**Открытые вопросы Ф3:**
- Способ установки скиллов Claude Code в headless-контейнере — главный
  блокер, решить до Ф4.

## Фаза 4. Research Pipeline (дни 6-12)

Цель: FEATURE доходит до техспеки. Самая тяжёлая фаза.

- [ ] Реализовать `run_research_pipeline` по шагам:
  - [ ] po-helper: БФТ с плейсхолдерами (критерии приёмки + UJM/интерфейсы/
        НФТ). Проверить, что `claude -p` пишет `.docx`.
  - [ ] gh-поиск связанных Issue.
  - [ ] Repowise: дозаполнение секций контекста. **Требует решения:
        как подключить MCP Confluence/Jira к headless worker'у.**
  - [ ] Blueprint: Service Blueprint + IDEF0 + "Несогласованности".
  - [ ] deb8flow: дебаты. **Требует установки deb8flow в Dockerfile +
        точного синтаксиса CLI.**
  - [ ] SA-helper: техспека, закрывает пункты "Несогласованностей".
- [ ] Все артефакты в ветку `research/issue-<n>`, лейбл `research:done`.
- [ ] Проверить durable execution: убить worker в середине → продолжение
      после рестарта.

**Чекпоинт Ф4:** FEATURE с лейблом `research-me` → полный пакет аналитики
(БФТ, Blueprint, дебаты, техспека) в git-ветке.

**Открытые вопросы Ф4 (все известны заранее, не новые):**
- Загрузка скиллов po-helper/SA-helper (общий с Ф3).
- Установка и флаги deb8flow.
- Плейсхолдеры в шаблоне po-helper: подтвердить, что реальный шаблон их
  поддерживает.
- MCP Confluence/Jira для Repowise в headless-среде.
- Источник OKR.

## Фаза 5. Передача в разработку (дни 12-16)

Цель: замкнуть на OpenHands + pr-agent.

- [ ] OpenHands как отдельный сервис (свой sandboxing, docker.sock,
      изолированный хост — НЕ в этом docker-compose).
- [ ] Реализовать `trigger_openhands_resolver`: передать пакет артефактов
      (БФТ + blueprint + техспека) как контекст.
- [ ] pr-agent на PR от OpenHands (независимый ревьюер).
- [ ] После merge — обновление `capabilities.md` (замыкание цикла:
      advisor узнаёт про новую фичу).

**Чекпоинт Ф5:** полный цикл Issue → PR → review → merge → capabilities.

## Риск, который стоит держать в голове весь проект

Экономика "claude -p на подписке" неприменима — используем GLM через z.ai
(pay-as-you-go, без лицензионной неоднозначности OAuth-токена Anthropic).
Это снимает риск, который был бы при опоре на подписку Claude в CI. Но
следить за квотами/биллингом z.ai при росте объёма Issue всё равно нужно.
