# План демонстрации — Issue Agent Service

> **Дата:** 2026-07-14
> **Статус:** Черновик к ревью
> **Репозиторий-цель демо:** `kibarik/po-helper` (58 open Issue на момент составления)

---

## 1. Цель и аудитория

Показать, что **сделано** в Issue Agent Service, отделив это от того, что
**спроектировано** и что **задокументировано, но не построено**. Демо честно
разделяет три слоя зрелости — иначе аудитория примет дизайн за работающий код.

Аудитория: владелец продукта (PO) + стейкхолдеры, оценивающие стратегическую
цель — «делегировать первичный разбор и аналитику улучшений po-helper через
Issues».

## 4. Демо-скрипт (3 акта)

### Акт 1 — Возможность требований (SA-helper FNR-пайплайн) · ~5 мин

**Тезис:** система сама выводит требования из кода, с якорями истины и без
галлюцинаций.

**Шаги:**
1. Открыть `sa_documentation/FNR/FNR_1/task.md` — постановка проблемы (тяжёлые
   стадии = заглушки), таблица «симптом → корень» с якорями `файл:строка`.
2. Открыть `sa_documentation/FNR/FNR_1/system_requirements.md` — показать §4
   (функциональные требования с критериями приёмки) и §8.1 «Якоря истины»
   (каждое утверждение → код-доказательство).
3. Открыть `validation.md` — многоуровневая проверка документа.

**Критерий приёмки акта:** каждое требование в СТ прослеживается до якоря
`файл:строка`; ни одного факта без источника.

---

### Акт 2 — Layer A live-триаж · ~10 мин

**Тезис:** автономный intake-конвейер реально прогоняет живой Issue po-helper и
**публикует настоящий вывод** (advisor-ответ + метка) в GitHub.

**Основной вариант (live, DRY_RUN off) — прогон на dedicated тест-Issue:**
```bash
make up                          # temporal + worker
make backfill-one issue=<TEST_N> # живой прогон одного Issue, реальные мутации
make logs                        # хвост логов воркера — видны стадии
```
Параллельно открыть:
- Temporal UI `http://localhost:8080` → workflow `issue-kibarik/po-helper-<TEST_N>`
  → граф стадий (prefilter → gate → classify → dup → priority);
- сам Issue в GitHub → **реальный** advisor-комментарий + метка `advisor:*`
  появляются вживую на глазах аудитории.

**Fallback A (если z.ai/воркер нестабильны) — реплей #92:**
```bash
gh issue view 92 -R kibarik/po-helper --json labels,comments
```
Уже опубликованный `advisor:consultation` + структурный ответ (Ситуация /
Варианты / «это консультация, открой FEATURE») — настоящий вывод прошлого прогона.

**Fallback B (полностью безопасно, без мутаций) — DRY_RUN smoke:**
`DRY_RUN=1` + `make backfill-one issue=83` — workflow проходит стадии, но в GitHub
ничего не пишет. Демонстрирует процесс без результата.

**Критерий приёмки акта:**
- Основной: workflow → `COMPLETED`; в тест-Issue появился реальный advisor-ответ
  и метка; в Temporal UI виден пройденный граф стадий.
- Fallback A: комментарий #92 демонстрирует корректный роутинг consultation.
- Мутации ограничены **только** dedicated тест-Issue — прочие 57 не тронуты.

### Акт 3 — Стратегический roadmap: consolidation как FNR-2 · ~7 мин

**Тезис:** ответ на исходную цель («объединить похожие + беклог») — отдельный
consolidation-этап, оформляемый через тот же FNR-пайплайн.

**Шаги:**
1. Показать проблему на живых данных: кластер #57–#113 в po-helper плотно
   связан вручную («Связано: #NNN»), а Layer A `duplicate_check` — попарный
   лексический, авто-закрывает при 0.85 → риск ложных слияний. Issue **#111**
   буквально требует нелинейного дедупа («функциональный дубль ≠ целевой»).
2. Показать дизайн (Approach A): fan-out извлечение solution-profile → reduce
   кластеризация по механизму решения (+#111 guard) → синтез объединяющего Issue
   (исходные = поставщики требований) → PR (человек апрувит, авто-close нет).
3. Позиционировать как **FNR-2**: `fnr-new-task → fnr-concept → fnr-debate →
   fnr-system-requirements → validate-doc`. Демо показывает, что новая
   возможность входит в продукт **через собственный механизм требований**
   (dogfooding).

**Критерий приёмки акта:** аудитория понимает, что consolidation — это roadmap,
не работающий код, и что путь его внедрения = FNR-2, а не самоделка сбоку.

## 5. Что НЕ демонстрируем (явно вне scope демо)

- **FNR-1 реализация** (тяжёлые стадии + skills delivery) — задокументирована в
  СТ `IAS-001`, но не построена (`run_research_pipeline`/`run_bug_pipeline` =
  `NotImplementedError`). Показываем только её **требования** (Акт 1), не работу.
- **Layer B** (webhook-автостарт на новых Issue) — установка описана в README, в
  демо не поднимаем.
- **OpenHands resolver** — отдельный сервис, вне scope.
- **Массовый `go-live` прогон** (реальные комментарии/закрытия по всем 58 Issue)
  — НЕ запускаем в демо: риск авто-закрытия по наивному дедупу (см. §6, #111).

---

## 6. Разрыв «документировано ≠ построено ≠ проектируется»

Демо честно показывает три несовпадающих набора:

- **Построено:** Layer A (Акт 2).
- **Документировано в СТ:** FNR-1 (не построено; показываем только требования).
- **Проектируется:** consolidation (Акт 3; в СТ ещё нет).

Стратегическая цель PO («объединить похожие + беклог») **не покрыта ни одним**
из построенных слоёв. Акт 3 закрывает разрыв, оформляя consolidation как FNR-2.

## 7. Риски демо и запасные варианты

| Риск | Симптом | Fallback |
|------|---------|----------|
| Воркер не поднят / z.ai ключ невалиден | `make backfill-one` стартует workflow, но `classify` падает | Перейти на Вариант B (реплей #92) + показать Temporal history |
| Temporal history по старым прогонам стёрта retention | В UI пусто, кроме свежего | Запустить `make backfill-one issue=<свежий>` вживую при аудитории |
| Случайная мутация GitHub | Комментарий/лейбл на реальном Issue | Держать `DRY_RUN=1`; `go-live` в демо не вызывать |
| Долгий LLM-прогон затягивает демо | Стадии медленные | Заранее прогреть один workflow до демо; на экране — уже `COMPLETED` |

## 8. Критерии успеха демо (общие)

1. Аудитория различает три слоя зрелости (построено / документировано /
   проектируется) — без иллюзии «всё работает».
2. Акт 2 показывает реальный автономный прогон на живом Issue po-helper.
3. Акт 1 демонстрирует нулевую галлюцинацию (каждый факт → якорь).
4. Акт 3 даёт понятный путь внедрения стратегической цели (FNR-2), а не обещание.
5. Мутации GitHub ограничены **только** dedicated тест-Issue Акта 2; прочие 57
   Issue за время демо не тронуты (массовый `go-live` не вызывался).
